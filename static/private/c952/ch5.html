5.2 Addition and Subtraction core

Addition: bit-by-bit, right to left, carries propagate left. Subtraction: negate the second operand (two's complement), then add. Word size is 64 bits (LEGv8/ARMv8).

ALUArithmetic Logic Unit. Hardware that does add, subtract, and logical ops (AND, OR).
OverflowResult cannot fit in the available word size (here, 64 bits).
Two's complementNegate by inverting bits and adding 1.

When overflow can / cannot occur

OpA signB signOverflow result sign (wrong)Example
A + B≥ 0≥ 0< 0 (Pos+Pos = Neg)10 + 20 = 30
A + B< 0< 0≥ 0 (Neg+Neg = Pos)−10 + −20 = −30
A − B≥ 0< 0≤ 0 (Pos−Neg = Neg)10 − (−20) = 30
A − B< 0≥ 0≥ 0 (Neg−Pos = Pos)−10 − 20 = −30

No overflow when adding operands of different signs (sum ≤ |larger operand|). No overflow in subtraction when operands have the same sign (because A−B = A + (−B) is then add of opposite signs).

Unsigned vs signed

  • Unsigned: overflow is normally ignored — used for memory addresses.
  • Signed: hardware sets the overflow condition code; software/compiler decides what to do.
  • C and Java ignore integer overflow; Ada and Fortran require notification.
  • LEGv8 ADD, ADDI, SUB can overflow.

Byte/halfword arithmetic on LEGv8

LEGv8 core has integer arithmetic only on full 64-bit words. To do byte/halfword math:

  • Load with LDURB / LDURH
  • Compute with ADD, SUB, MUL, DIV, then AND the result to mask to 8 or 16 bits.
  • Store with STURB / STURH.

Saturating arithmetic (clamps at max/min instead of wrapping) is used in multimedia extensions, not in general-purpose integer ops. Carry-lookahead anticipates carries to speed addition (log-time worst case).

5.3 Multiplication core

Paper-and-pencil shift-and-add. Operand 1 = multiplicand, operand 2 = multiplier, result = product. An n-bit × m-bit multiply needs n + m bits for the product.

MultiplicandFirst operand. Held in left input of ALU; gets shifted left each step (in 1st HW version).
MultiplierSecond operand. Examine least significant bit, then shift right.
ProductResult, n+m bits wide. 128 bits for two 64-bit operands.

First sequential algorithm — 3 steps per bit, 64 iterations

  1. If LSB of Multiplier = 1: Product ← Product + Multiplicand. If 0: skip.
  2. Shift Multiplicand register left 1 bit.
  3. Shift Multiplier register right 1 bit.

Repeat 64 times. ~200 clock cycles if each step takes 1 cycle.

Register widths

Register1st versionRefined version
Multiplicand128-bit64-bit
ALU128-bit64-bit
Multiplier64-bit (separate)placed in right half of Product
Product128-bit128-bit, shifted right each step

Faster (parallel-tree) multiply

Use 63 adders organized as a tree (and carry-save adders). Time goes from ~64 add times down to ~log₂(64) = 6 add times. Pipelinable.

Signed multiplication

Convert operands to positive, run algorithm, then negate the product if the original signs disagree. Or: sign-extend during shifts.

LEGv8 multiply instructions

InstrPurpose
MULLower 64 bits of 128-bit product (signed or unsigned look the same in low half).
SMULHUpper 64 bits, signed.
UMULHUpper 64 bits, unsigned.

LEGv8 multiply does NOT set the overflow condition code. Software checks: no overflow iff upper 64 = 0 (UMULH) or upper 64 = replicated sign bit of lower 64 (SMULH).

Compilers replace multiply by power of 2 with a left shift (strength reduction).

5.4 Division core

Long division: try to subtract divisor from a portion of the dividend, generate quotient bit, bring down next bit.

DividendThe number being divided. Initially loaded into the Remainder register.
DivisorNumber you divide by. Starts in left half of 128-bit Divisor register; shifts right each iteration.
QuotientPrimary result. Q × Divisor + Remainder = Dividend.
RemainderSecondary result. |Remainder| < |Divisor|. Sign matches the dividend.
Quotient × Divisor + Remainder = Dividend (Remainder < Divisor in magnitude)

Restoring division algorithm — needs n + 1 = 65 iterations

  1. Subtract Divisor from Remainder, store result in Remainder.
  2. If Remainder < 0: 2b — restore by adding Divisor back; shift Quotient left, place 0 in LSB.
    If Remainder ≥ 0: 2a — keep new Remainder; shift Quotient left, place 1 in LSB.
  3. Shift Divisor register right 1 bit. Repeat 65 times.

Improved hardware halves the ALU/Divisor width to 64 bits and merges the Quotient register into the right half of the Remainder register (which then shifts left).

Signed division

  • Negate the quotient if the signs of operands disagree.
  • Sign of remainder must match sign of dividend.

Faster division

You can't use the multiply-style parallel tree, because the next step depends on the sign of the previous difference. SRT division uses a lookup table (typically 6 bits of remainder, 4 bits of divisor) to predict ~4 quotient bits per step and corrects mispredictions later. Newton's iteration computes 1/c then multiplies.

LEGv8 divide instructions

InstrMeaning
SDIVSigned divide.
UDIVUnsigned divide.

LEGv8 divide ignores overflow and division by 0. Software must check the divisor before dividing. Same sequential hardware can do both multiply and divide (128-bit shift register + 64-bit ALU).

5.5 Floating Point core / heavily tested

Floating point = computer arithmetic where the binary point is not fixed. C type float = single precision (32 bits); double = double precision (64 bits).

Scientific notationOne nonzero digit left of the (binary) point.
NormalizedScientific notation with no leading zeros.
Fraction (mantissa)Bits to the right of the binary point. 23 bits single, 52 bits double.
ExponentBiased power-of-2 scale. 8 bits single, 11 bits double.
Significand1 + fraction (the implicit leading 1 + the fraction bits). 24 single, 53 double.
Sign (S)1 = negative, 0 = positive. Always the most significant bit.
Overflow (FP)Positive exponent too large to fit in exponent field.
Underflow (FP)Negative exponent too large in magnitude to fit.
Exception / InterruptUnscheduled "procedure call" used to signal overflow/underflow/etc.

IEEE 754 layout

Value = (−1)^S × (1 + Fraction) × 2^(Exponent − Bias) Single: bias = 127 Double: bias = 1023

Special exponent values (single precision biased)

Why bias instead of two's complement for the exponent?

So FP values can be sorted/compared with integer comparison instructions. Layout puts sign first, exponent next, fraction last — for the same sign, larger exponent ⇒ larger number ⇒ larger as an unsigned bit pattern.

Range (approximate)

FormatSmallest normalLargest finite
Single~2.0 × 10⁻³⁸~2.0 × 10³⁸

Worked example: encode −0.75 single precision

  1. −0.75₁₀ = −3/4 = −0.11₂ = −1.1₂ × 2⁻¹
  2. S = 1; Exponent (biased) = −1 + 127 = 126 = 01111110; Fraction = 10000000000000000000000
  3. Result: 1 01111110 10000000000000000000000

Worked example: decode 1 10000001 01000…0 single precision

  1. S = 1 ⇒ negative.
  2. Exponent field = 10000001 = 129; true exponent = 129 − 127 = 2.
  3. Fraction = 0.01₂ = 0.25 ⇒ significand = 1.25.
  4. Value = −1 × 1.25 × 2² = −5.0

Floating-point addition (4 steps)

  1. Align: shift the significand of the number with the smaller exponent right until exponents match.
  2. Add the significands.
  3. Normalize the sum (shift left or right; adjust exponent). Check overflow / underflow.
  4. Round. If rounding un-normalizes, repeat step 3.

FP addition is NOT associative. Order of operations affects the result.

Floating-point multiplication (5 steps)

  1. Add the exponents. Using biased exponents, subtract the bias once: e₁ + e₂ − bias.
  2. Multiply the significands.
  3. Normalize. Check overflow / underflow.
  4. Round. Re-normalize if needed.
  5. Set sign: XOR of operand signs (different signs ⇒ negative).

FP register banks: S0..S31 (single, lower half of D registers), D0..D31 (double). Address registers for FP loads/stores are still integer registers (X-registers). Branch on FP compare uses B.cond.

Example: float f2c(float fahr) returning (5.0/9.0)*(fahr − 32.0)

f2c:
    LDURS S16, [X27,const5]    // S16 = 5.0
    LDURS S18, [X27,const9]    // S18 = 9.0
    FDIVS S16, S16, S18        // S16 = 5.0/9.0
    LDURS S18, [X27,const32]   // S18 = 32.0
    FSUBS S18, S12, S18        // S18 = fahr - 32.0
    FMULS S0,  S16, S18        // S0  = (5/9)*(fahr-32.0)
    BR    LR                   // return

Rounding: guard, round, sticky bits

GuardFirst extra bit kept right of the significand during intermediate calc.
RoundSecond extra bit. Together with guard, lets you round to the nearest representable.
Sticky bitSet to 1 if any nonzero bit is shifted off past the round bit. Lets HW distinguish ".5000…01" from ".5000".
  • IEEE 754 has 4 rounding modes: round up (toward +∞), round down (toward −∞), truncate, round to nearest even (default; the only one Java supports).
  • Round-to-nearest-even: in halfway tie, force the LSB to 0 (creates a 0 in least significant retained bit).
  • IEEE 754 guarantees results within ½ ulp of the true value (no overflow/underflow/invalid).

Other terms

ulp (units in the last place)Number of bits of error in the LSBs of the significand.
Fused multiply-add (FMA)a = a + (b × c) with one rounding (after the add). Higher precision than separate mul + add. Added to IEEE 754-2008. ARMv8 supports it.
Denormal / subnormalExponent = 0, fraction ≠ 0. Allows gradual underflow between 0 and the smallest normalized number.
Row-major vs column-majorC/Java store arrays row-major; Fortran column-major.

5.6 Subword Parallelism core

Partition a wide adder's carry chains so the same instruction operates on multiple smaller values in parallel. Cheap to implement, large speedups for graphics/audio.

Also called data-level parallelism, vector, or SIMD (Single Instruction, Multiple Data).

ARMv8 SIMD

  • 32 new 128-bit registers: V0–V31.
  • 500+ SIMD machine instructions (245 distinct assembly mnemonics).
  • Supports 8/16/32/64/128-bit signed/unsigned ints, plus 32-bit and 64-bit floats.

Width suffixes (after the V-register name)

Ssingle (32-bit)4

Prefixes on SIMD mnemonics: U unsigned, S signed, F floating-point, P polynomial, none = raw bits.

Naming conventions

  • SS = scalar single (1 × 32-bit FP); PS = packed single (4 × 32-bit FP).
  • SD = scalar double (1 × 64-bit FP); PD = packed double (2 × 64-bit FP in XMM, 4 in YMM).
  • AVX prefixes v on SSE2 mnemonics and uses YMM regs; also enables 3-address form (e.g. vaddpd %ymm0,%ymm1,%ymm4) instead of the 2-address SSE form.

5.8 Rest of ARMv8 Arithmetic

  • 63 assembly-language integer + FP arithmetic instructions in full ARMv8.
  • 15 ARMv8 arithmetic core instructions.
  • 245 SIMD assembly instructions; 18 FP data transfer instructions in machine language.
  • Notable extras: MNEG (multiply-negate), FNMUL (FP multiply-negate), FCCMP (FP conditional compare; only compares if initial cond is true), quiet vs signaling NaN compares, FCSEL.
  • SIMD width modifiers: Wide, Long, Narrow, Pair. Suffix 2 = work on upper half of register.

5.10 Fallacies and Pitfalls exam-likely

Fallacy: right shift = divide by 2ⁿ

  • True for unsigned integers.
  • False for signed with logical right shift (zero-fill) — the sign bit is destroyed.
  • An arithmetic right shift (sign-extends) gets close, but rounds toward −∞: e.g., −5 ASR 1 = −3, not −2 (truncated divide).

Pitfall: floating-point addition is not associative

(a + b) + c ≠ a + (b + c) in general. Example: a = −1.5e38, b = 1.5e38, c = 1.0. (a + b) + c = 0 + 1 = 1.0; a + (b + c) = a + (1.5e38) = 0.0.

Fallacy: parallel and sequential FP code give the same answer

Because FP add is non-associative, summing N numbers can give different answers depending on the number of processors / order of partial sums. Integer two's-complement addition is associative (even on overflow), so this fallacy applies only to FP.

Fallacy: only theoretical mathematicians care about FP accuracy

1994 Pentium FDIV bug: bad SRT lookup table entries produced wrong bits 12–52 of some divides. Cost Intel ~$500 million in recall.

5.11 Concluding Remarks

  • Two's complement integer arithmetic is universal.
  • IEEE 754 is the standard for floating-point representation and computation.
  • Numerical analysis is the field that handles imprecision and limited representation.
  • Bit patterns have no inherent meaning. The instruction acting on them determines whether they're signed/unsigned ints, FP, instructions, characters, etc.
  • Underflow = magnitude smaller than smallest representable; overflow = larger than largest.

Quick-reference cheat sheet last 10 minutes

IEEE 754 quick numbers

  • Single: 1 + 8 + 23, bias 127, sig 24 bits
  • Double: 1 + 11 + 52, bias 1023, sig 53 bits
  • Half: 1+5+10, bias 15. Quad: 1+15+112, bias 16383.
  • Value = (−1)^S × (1 + frac) × 2^(exp − bias)
  • Exp = 0 ⇒ ±0 or denormal; Exp = all 1s ⇒ ±∞ or NaN
  • Sticky / round / guard bits for accurate rounding
  • Round-to-nearest-even is default and only mode Java supports
  • Result accuracy: within ½ ulp

Overflow rules (signed integer)

  • Add two pos → got neg ⇒ overflow
  • Add two neg → got pos ⇒ overflow
  • Sub neg from pos → got neg ⇒ overflow
  • Sub pos from neg → got pos ⇒ overflow
  • Different signs in add (or same signs in sub): can't overflow
  • Unsigned: hardware ignores; used for addresses

Multiply / divide register widths

  • Sequential MUL: 128-bit Mcand+ALU+Product, 64-bit Mplier; refined: 64-bit Mcand+ALU, 128-bit Product (with multiplier in low half), shift right
  • Sequential DIV: 128-bit Divisor+ALU+Remainder, 64-bit Quotient; needs n+1 = 65 iterations (restoring)

LEGv8 arithmetic mnemonics

  • Integer: ADD ADDI SUB MUL SMULH UMULH SDIV UDIV
  • FP single: FADDS FSUBS FMULS FDIVS FCMPS LDURS STURS
  • FP double: FADDD FSUBD FMULD FDIVD FCMPD LDURD STURD
  • FP regs: S0..S31 single (low half of D0..D31)
  • Branch on FP cond: B.cond
  • MUL doesn't set overflow; DIV ignores overflow & div-by-0

SIMD lane suffixes (ARMv8 V regs, 128-bit)

  • 16B / 8H / 4S / 2D / 1Q
  • 32 V-registers, 245 SIMD instructions
  • Prefix U/S/F = unsigned/signed/floating

x86 SSE/AVX

  • SS=1×32, PS=4×32, SD=1×64, PD=2×64 (XMM) / 4×64 (YMM)
  • XMM=128, YMM=256, ZMM=512
  • AVX = SSE2 ops with leading v, 3-address form
  • DGEMM AVX speedup ≈ 3.85× over scalar

Rules of thumb

  • Multiply by 2ⁿ ⇒ shift left (strength reduction)
  • Divide unsigned by 2ⁿ ⇒ shift right; signed needs ASR (still rounds toward −∞)
  • FMA = one rounding ⇒ more accurate than mul + add separately
  • FP add is NOT associative; integer 2's-comp add IS
  • Sign of remainder matches sign of dividend

Self-test (likely exam-style Q&A)

Integer addition / subtraction / overflow
How does a computer perform subtraction?
Negate the second operand (two's complement: invert bits and add 1) and add.
When can overflow NOT occur in addition?
When the two operands have different signs. The sum's magnitude can't exceed the larger operand.
Two positive numbers added give a negative result. What happened?
Overflow — the sign bit got set with a value bit because the result needed 65 bits.
Does ADD set the overflow condition code on LEGv8?
Yes. ADD, ADDI, and SUB can set the overflow condition code. C and Java ignore it; Ada and Fortran require notification.
Does overflow occur in unsigned addition?
Mathematically yes, but it's normally ignored — unsigned ints are typically used for memory addresses, where wrap-around is acceptable.
What is an ALU?
Arithmetic Logic Unit — hardware that performs add, subtract, and usually AND/OR.
How do you do byte/halfword arithmetic on LEGv8?
Load with LDURB/LDURH, compute with full-word ADD/SUB/MUL/DIV, mask the result with AND to 8 or 16 bits, store with STURB/STURH.
Multiplication
What's the largest possible product of an n-bit and m-bit value?
n + m bits long.
In the first sequential multiply hardware, how wide is each register?
Multiplicand 128, ALU 128, Product 128, Multiplier 64.
In the refined hardware, what changed?
Multiplicand and ALU shrink to 64 bits; Multiplier register disappears (placed in right half of Product); Product is shifted right each step.
How many iterations does the basic sequential algorithm take?
64 (one per bit of the multiplier).
How many add-times does the parallel-tree multiplier take?
log₂(64) = 6 add-times (instead of 64).
Why can the same hardware do both multiply and divide?
Both need a 128-bit shift register and a 64-bit ALU that adds or subtracts.
Which instruction returns the upper 64 bits of a signed 128-bit product?
SMULH. UMULH for unsigned. MUL gives the lower 64 bits.
Does MUL detect overflow?
No. Software must check: no overflow iff the high 64 bits are 0 (UMULH) or replicated sign of the low 64 (SMULH).
What's "strength reduction" in multiplication?
Compiler replaces multiply by a power of 2 with a left shift.
Division
Quotient × Divisor + ___ = Dividend
Remainder. And |remainder| < |divisor|.
In restoring division hardware, where is the divisor placed initially?
In the left half of the 128-bit Divisor register; shifted right 1 bit per iteration.
If the remainder goes negative after subtraction, what happens?
Restore: add the divisor back; shift the quotient left and place a 0 in its LSB.
If the remainder is non-negative, what happens?
Keep the new remainder; shift the quotient left and place a 1 in its LSB.
How many iterations does the algorithm need?
n + 1 = 65 for 64-bit operands.
Why can't you parallelize division like multiplication?
Each step depends on the sign of the previous difference (you can't precompute everything).
What is SRT division?
A faster algorithm that predicts multiple quotient bits per step (typically 4) using a lookup table indexed by upper bits of remainder and divisor; corrects mispredictions later.
Which sign does the remainder take?
The same sign as the dividend, regardless of the divisor's sign.
Do SDIV and UDIV detect overflow or division by 0?
No — software must check the divisor before dividing.
Floating point — representation
Single precision field widths?
1 sign + 8 exponent + 23 fraction = 32 bits. Bias 127. Significand = 24 bits including implicit leading 1.
Double precision field widths?
1 + 11 + 52 = 64 bits. Bias 1023. Significand = 53 bits.
What is "normalized" form?
Exactly one nonzero bit (a 1) to the left of the binary point. IEEE 754 makes it implicit.
Why bias the exponent instead of using two's complement?
So FP values can be sorted/compared with integer comparison instructions: smallest exponent maps to smallest unsigned bit pattern.
What does an exponent of all 0s with a zero fraction represent?
±0 (zero is special-cased — no implicit leading 1).
All 1s exponent + zero fraction?
±∞.
All 1s exponent + nonzero fraction?
NaN (Not a Number) — used for invalid ops like 0/0 or ∞−∞.
What is a denormal/subnormal?
Exp = 0, fraction ≠ 0. Allows gradual underflow between 0 and the smallest normalized number.
FP overflow vs FP underflow?
Overflow = positive exponent too large for the field. Underflow = negative exponent too large in magnitude (number too small).
Encode −0.75 in single precision.
−0.75 = −1.1₂ × 2⁻¹. S=1, exp=126=01111110, frac=10000…0. Bits: 1 01111110 10000000000000000000000.
What decimal value is 1 10000001 01000…0 in single precision?
−1 × 1.25 × 2² = −5.0.
Why is the implicit leading 1 worth doing?
It buys an extra bit of precision for free (24 significand bits in 32 total).
Floating point — arithmetic and rounding
Steps of FP addition?
1) Align (shift smaller exponent's significand right to match), 2) add significands, 3) normalize and check overflow/underflow, 4) round (re-normalize if needed).
Steps of FP multiplication?
1) Add exponents (subtract one bias if biased), 2) multiply significands, 3) normalize, 4) round, 5) set sign (XOR of operand signs).
When you add biased exponents, why do you subtract the bias?
Because each operand was already biased, so the sum has the bias added twice; subtracting once gets back to single-biased form.
What are guard, round, and sticky bits?
Three extra bits kept right of the significand. Guard + round (2 bits) let you round to nearest representable. Sticky records whether any nonzero bit shifted off past round, to break exact-half ties correctly.
What are IEEE 754's four rounding modes?
Round up (toward +∞), round down (toward −∞), truncate, round to nearest even (default).
In round-to-nearest-even, how do you break a tie?
Round so that the LSB of the retained significand is 0 (even).
What is one ulp?
"Unit in the last place" — the gap between adjacent representable values; an error of N ulps means N bits off in the LSBs of the significand.
What is fused multiply-add?
a = a + (b×c) computed with one rounding (after the add) instead of two — more precise than separate FMUL + FADD. Added to IEEE 754-2008.
LEGv8 / ARMv8 floating-point instructions
FP add single vs double?
FADDS for single, FADDD for double.
Why does LEGv8 have separate FP registers (S/D) instead of using X-registers?
Doubles register count without expanding instruction format, doubles register bandwidth, allows internal-format customization, and historically allowed FP unit on a separate coprocessor chip.
Are address registers for FP loads/stores integer or FP registers?
Integer X-registers. Only the data ends up in S/D registers.
Relationship between S and D registers?
Sn is the lower half of Dn.
How do you branch on a floating-point comparison?
Use FCMPS/FCMPD to set condition codes, then B.cond — same branch instruction as integer.
SIMD / subword parallelism / SSE/AVX
What is subword parallelism?
Partitioning a wide adder so a single instruction performs several smaller arithmetic operations in parallel.
How wide are ARMv8 V-registers, and how many are there?
128 bits, 32 of them (V0..V31).
What does ADD V1.16B, V2.16B, V3.16B do?
16 parallel 8-bit integer adds.
What does FADD V1.4S, V2.4S, V3.4S do?
4 parallel 32-bit floating-point adds.
XMM vs YMM vs ZMM widths?
XMM 128-bit, YMM 256-bit, ZMM 512-bit.
What do PS, PD, SS, SD mean in SSE/AVX names?
PS = packed single (4×32-bit FP in XMM), PD = packed double (2×64-bit), SS = scalar single, SD = scalar double.
What's the difference between SSE2 addpd and AVX vaddpd?
vaddpd uses YMM (256-bit, 4 doubles instead of 2) and accepts 3 address operands instead of 2.
What was the AVX speedup on 32×32 DGEMM vs unoptimized?
3.85×, close to the theoretical 4× from doing 4 doubles in parallel.
What does _mm256_broadcast_sd do?
Replicates one scalar double 4 times across a YMM register — used to multiply each lane of A by the same B element.
Fallacies and pitfalls
Is right-shift always the same as integer division by a power of 2?
No. True for unsigned. For signed, logical shift right corrupts the sign bit; arithmetic right shift sign-extends but rounds toward −∞ (e.g., −5 ASR 1 = −3, not −2).
Is floating-point addition associative?
No. (a + b) + c can differ from a + (b + c) due to limited precision.
Is two's-complement integer addition associative?
Yes — even when overflow occurs.
Why might the same FP code give different answers on 10 vs 1000 processors?
Because partial sums are added in different orders, and FP add is not associative.
What was the 1994 Pentium FDIV bug?
5 bad lookup-table entries in the SRT divider produced wrong bits 12–52 in some divides. Cost Intel ~$500M to recall.

Common exam traps

  • "The single-precision exponent is 7 bits." No. 8 bits exponent, 23 fraction, bias 127. Double = 11 / 52 / 1023.
  • "The implicit leading 1 makes the significand 23 bits." No. 23 fraction bits + 1 implicit = 24-bit significand (single). 53-bit (double).
  • "Adding two negatives can never overflow." No. Two negatives can overflow to a positive (e.g., −10 + −20 wrapping to +). Two operands of different signs can't overflow in addition.
  • "Right shift always equals divide by 2ⁿ." No. Only for unsigned. Signed needs arithmetic shift, and even that rounds toward −∞ rather than toward zero.
  • "FP add and integer add are both associative." No. Only integer two's-complement add is associative; FP is not.
  • "MUL on LEGv8 sets the overflow condition code." No. Multiply does not set overflow; software must check via UMULH/SMULH.
  • "SDIV / UDIV trap on division by 0." No. They ignore overflow and divide-by-0; software must check the divisor.
  • "The remainder takes the divisor's sign." No. The remainder's sign matches the dividend.
  • "NaN is the same as ±∞." No. Both use exponent = all 1s, but ∞ has zero fraction; NaN has nonzero fraction.
  • "Exponent stored in two's complement." No. Exponent uses biased notation (subtract bias to get true exponent) so values sort with integer comparisons.
  • "The base register for an FP load is an S or D register." No. The base/address register is still an integer X-register; only the data ends up in S/D.
  • "The first sequential multiply HW iterates 32 times." No. 64 times for 64-bit operands. Division: 65 (n+1).
  • "Parallel-tree multiply takes 64 add-times." No. log₂(64) = 6 add-times. The 64-iteration count is the sequential algorithm.
  • "Round to nearest even rounds 0.5 always up." No. Halfway ties round to whichever neighbor has an even LSB (so half the time up, half down).
  • "FMA is just FMUL then FADD." No. FMA does one rounding after the add, gaining precision over separate FMUL + FADD.
  • "AVX gave 4× speedup exactly on DGEMM." Close — but the textbook value is 3.85×, not 4.0× (memory and overhead reduce the theoretical max).
  • "All bit patterns have a fixed meaning." No. The instruction operating on the bits determines whether they're a signed int, unsigned int, FP, instruction, character, etc.
