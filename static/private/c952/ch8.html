8.1 Introduction core

Energy is now the dominant constraint, so future performance increases come from explicit hardware parallelism rather than higher clock rates. Sequential code = slow code today.

Core Definitions

MultiprocessorA computer with at least two processors. Uniprocessors are increasingly rare.
Multicore microprocessorA microprocessor with multiple processors ("cores") on one IC. Cores are typically Shared Memory Processors (SMPs).
Task / process-level parallelismRun independent single-threaded apps simultaneously. Different from running ONE program in parallel.
Parallel processing programA SINGLE program that runs on multiple processors simultaneously.
ClusterSet of computers connected over a LAN, functioning as a single large multiprocessor (message-passing).
Shared memory multiprocessor (SMP)Parallel processor with a single physical address space. Almost all multicore chips are SMPs.

Sequential vs Concurrent vs Serial vs Parallel key

Sequential SWConcurrent SW
Serial HWMATLAB matrix-multiply on Pentium 4OS on a uniprocessor
Parallel HWMATLAB matrix-multiply on Core i7 (auto-parallelized)OS on a multicore

Two axes — software is sequential or concurrent; hardware is serial or parallel. The parallel revolution challenge: make both kinds of software fast on parallel hardware.

8.2 The Difficulty of Creating Parallel Processing Programs core

The hardware is not the hard part — getting parallel software to actually run faster is. Challenges:

  • Scheduling, partitioning into balanced pieces.
  • Load balancing (one slow processor wastes the others).
  • Synchronization overhead.
  • Communication overhead.
  • Amdahl's Law — even small sequential portions limit total speed-up.

Amdahl's Law (re-formulation)

Speed-up = 1 / [ (Fraction_affected / Improvement) + (1 - Fraction_affected) ]

Example: To get 90× speed-up with 100 processors, the sequential portion must be ≤ 0.1%.

Strong vs Weak Scaling key

Strong scalingSpeed-up while keeping problem size FIXED. Limited by Amdahl's Law for large N.
Weak scalingProblem size grows proportionally with the number of processors. Often easier — many real workloads naturally scale this way (TPC-C, Linpack).

Memory hierarchy can flip the conventional wisdom: if a weakly-scaled dataset no longer fits in last-level cache, performance collapses.

Load Balance

40-processor example with sum of 400 additions: if one processor has 5% of the load (20 ops while others share 380/39 ≈ 9.7 each), max time = 20×t. Speed-up drops from 20.5 to ~10. With 12.5% load on one processor → speed-up drops to ~8 — others idle >80% of the time.

8.3 SISD, MIMD, SIMD, SPMD, and Vector core

Flynn's Taxonomy

AcronymMeaningExample
SISDSingle Instruction stream, Single Data streamConventional uniprocessor
SIMDSingle Instruction stream, Multiple Data streamsVector processor; x86 SSE/AVX subword parallelism
MISDMultiple Instruction streams, Single Data stream"Stream processor" — pipelined ops on a single stream (rare)
MIMDMultiple Instruction streams, Multiple Data streamsMulticore, cluster, multiprocessor

SPMD key

Single Program, Multiple Data streams. The conventional way to program a MIMD: write one program, all processors run it, branches decide which processor does what.

Why SIMD Is Useful

  • All execution units synchronized to one PC → simpler programming, close to SISD.
  • Reduced instruction bandwidth: one copy of code, not n copies.
  • Best when there's lots of identically-structured data ("data-level parallelism").
  • Worst on case/switch statements — each unit needs different logic; the wrong-data units are disabled, so SIMD effectively runs at 1/n speed.

SIMD in x86

MMX (1996) → SSE → SSE2 → AVX → AVX2. AVX supports four 64-bit FP ops simultaneously. Width and op are encoded in the opcode → hundreds of opcodes accumulated over time.

Vector Architecture key

Older, more elegant SIMD style (Cray, 1970s). Pipelines an ALU rather than building 64 parallel ALUs. Uses vector registers (e.g., 32 vector regs × 64 64-bit elements).

DAXPY Example: Scalar vs Vector

Conventional LEGv8 DAXPY: ~10-instruction inner loop × 64 iterations ≈ 600 dynamic instructions. Vector LEGv8 DAXPY: 6 instructions total. Bonus: pipeline stalls happen once per vector op, not once per element.

// Scalar (per iteration)
LDURD D2,[X19,#0]
FMULD D2,D2,D0
LDURD D4,[X20,#0]
FADDD D4,D4,D2
STURD D4,[X20,#0]
ADDI  X19,X19,#8 / ADDI X20,X20,#8 / loop branch

// Vector (whole loop)
LDURD   D0,[x28,a]
LDURDV  V1,[X19,#0]
FMULDVS V2,V1,D0
LDURDV  V3,[X20,#0]
FADDDV  V4,V2,V3
STURDV  V4,[X20,#0]

Vector vs Scalar — Why Vectors Win

  • One instruction = an entire loop's worth of work; instruction fetch/decode bandwidth drops dramatically.
  • Compiler/programmer asserts independence within a vector → hardware skips intra-vector hazard checks.
  • Hazard checking once per vector operand, not per element.
  • Memory access pattern is known (contiguous, strided, or indexed/gather-scatter). Latency is amortized over the vector.
  • No control hazard from a loop branch.
  • Energy-efficient.

Vector vs Multimedia Extensions

VectorMultimedia (SSE/AVX)
Operation count per instructionDozens (set in a separate length register)A few (encoded in opcode)
Length compatibilityBinary-compatible across implementationsNew opcodes for each width (MMX→SSE→AVX)
Memory access patternsContiguous, strided, indexed (gather-scatter)Mostly contiguous; SSE requires 16-byte alignment

Vector Lanes

A vector unit may have multiple parallel lanes, each containing functional units + a slice of the vector register file. Going from 1 lane → 4 lanes reduces clocks per vector instruction by ~4×. Per-lane register storage holds every Nth element of each vector register.

Strip Mining

When a loop is longer than the maximum vector length, the compiler emits bookkeeping code to iterate full-length vector ops plus a leftover handler.

8.4 Hardware Multithreading core

Lets multiple threads share the functional units of one processor in an overlapping way to improve utilization. The processor must duplicate per-thread state (PC, register file). Memory shared via standard virtual memory.

Thread vs Process

ThreadPC, register state, stack. A lightweight process. Shares an address space with sibling threads. Thread switch can be near-instantaneous.
ProcessOne or more threads + address space + OS state. Process switch invokes the OS — hundreds to thousands of cycles.

Three Hardware-Multithreading Styles key

StyleWhen to switchProsCons
Fine-grainedEvery clock cycle (round-robin, skipping stalled threads)Hides short and long stallsSlows execution of an individual ready thread
Coarse-grainedOnly on expensive stalls (e.g., last-level cache miss)Doesn't slow a thread when no stall; cheap thread switchPipeline must be drained/refilled at switch → start-up cost; bad at hiding short stalls
Simultaneous Multithreading (SMT)Never — multiple threads issue instructions in the SAME cycle on a multi-issue, dynamically-scheduled superscalarExploits ILP and TLP simultaneously; uses dynamic scheduling that already existsAdds hardware complexity, register-renaming pressure

Intel Core i7 has 2-thread SMT per core. Average speed-up from SMT on PARSEC: ~1.31×, with ~1.07× energy improvement.

8.5 Multicore and Other Shared Memory Multiprocessors core

SMP = Shared Memory Processor. All processors share a single physical address space. Hardware typically provides cache coherence. Programs communicate through shared variables.

UMA vs NUMA key

UMA — Uniform Memory AccessMemory access latency is the same regardless of which processor and which word. Easier to program. Limits scalability.
NUMA — Non-Uniform Memory AccessSome memory accesses are faster than others (memory is divided across processors). Harder to program but scales to more processors and offers low local-memory latency.

Synchronization

Shared address spaces need coordination; one approach is a lock — only one processor at a time may hold it. Other processors wait. ARMv8 has lock instructions covered earlier.

Reduction Pattern

To sum 64,000 numbers on 64 processors:

  1. Each processor sums 1,000 of them in parallel into sum[Pn].
  2. Hierarchical reduction: 32 processors add pairs of partial sums → 16 → 8 → 4 → 2 → 1, with barrier synchronization between rounds.

OpenMP key

Portable API for shared memory parallelism in C/C++/Fortran. Built into most C compilers (cc -fopenmp foo.c). Uses pragmas:

#pragma omp parallel num_threads(P)        // set thread count
#pragma omp parallel for                   // parallelize a for loop
#pragma omp parallel for reduction(+:sum)  // parallel reduction

8.6 Introduction to GPUs core

GPUs distinguished themselves from CPUs by following the game industry's drive for high graphics performance. Today they're widely used for compute too.

How GPUs Differ from CPUs

  • GPUs are accelerators — they supplement a CPU, don't replace it.
  • Problem sizes: hundreds of MB to a few GB (vs CPUs with hundreds of GB).
  • No multilevel caches to hide latency — uses massive hardware multithreading instead.
  • Memory oriented for bandwidth, not latency — graphics DRAM (GDDR) with wide buses.
  • More processors, each one more multithreaded than a CPU core.
  • For general compute, you must include CPU↔GPU memory transfer time.

NVIDIA Fermi-style GPU Vocabulary key

Two Levels of Hardware Schedulers

  1. Thread Block Scheduler: assigns blocks of threads to SMs.
  2. SIMD Thread Scheduler: inside an SM, picks which SIMD threads run when (fine-grained multithreading).

Fermi Specifics

  • 32-wide SIMD threads mapped to 16 SIMD lanes → each SIMD instruction takes 2 clock cycles.
  • Each SM has 32,768 32-bit registers; each SIMD thread limited to 64 registers.
  • Each lane: 2048 registers. Each CUDA thread = one element on one lane.
  • 4 implementations: 7, 11, 14, or 15 multithreaded SIMD processors.

CUDA & OpenCL

NVIDIA's CUDA = C-inspired language for writing GPU programs directly, bypassing graphics APIs. OpenCL = multi-vendor portable equivalent.

8.7 Clusters & Warehouse-Scale Computers core

Message-Passing Multiprocessors

  • Each processor has its own private physical address space.
  • Communicate via send_message and receive_message routines.
  • Coordination is built-in: sender knows when sent, receiver knows when received.
  • Easier hardware, harder programming (must identify all communication explicitly).
  • Better for task-level parallelism with little communication (web search, mail, file servers).

Clusters

  • Set of computers connected over Ethernet → message-passing multiprocessor.
  • Each node runs its own OS copy.
  • Lower communication performance than SMP, but cheaper, more available, easier to expand and replace failed nodes.
  • Powers Internet services (Amazon, Facebook, Google, Microsoft).

Warehouse-Scale Computers (WSCs) key

Buildings housing 50,000–100,000 servers, designed as one giant computer. Cost ~$150M for building, power, cooling, servers, networking. Three distinguishing characteristics vs traditional servers:

  1. Ample, easy parallelism: batch (MapReduce/Hadoop) and interactive (SaaS) workloads have huge request-level parallelism with few synchronization needs.
  2. Operational costs count: 10+ year lifespans → energy/power/cooling is >30% of TCO.
  3. Scale opportunities AND problems: volume discounts → cloud computing is profitable. But with 100K servers and 25-year MTBF, you still see ~5 server failures per day. With 4 disks/server at 2% AFR, ~1 disk fails per hour. Fault tolerance is essential.

MapReduce / Hadoop

Inspired by Lisp. Map: programmer-supplied function processes each input record, emitting (key, value) pairs. Reduce: collects values per key and collapses with another programmer function. Highly parallel, easy to use.

SaaS & Cloud Computing

Software run at remote site, available via Internet (typically web). Customers pay for use, not ownership. Cloud's economy of scale lets cloud companies rent servers at a profit while undercutting the customer's own infrastructure cost.

Grid Computing

Distributed over wide areas; PCs harvest idle cycles. SETI@home: 5M+ users, 200+ countries; ~3.5 PetaFLOPS by end of 2011 — faster than the best 2013 supercomputer.

8.8 Multiprocessor Network Topologies

Performance Metrics key

Total network bandwidthSum of all link bandwidths. Best-case (peak) metric. For a ring with P processors and link bandwidth B: total = P × B.
Bisection bandwidthSum of bandwidth of links crossing the worst-case bisection of the network. For a ring: 2 × B. For a bus: 1 × B. For a fully connected network: (P/2)² × B.

Common Topologies

TopologyTotal bandwidth (P nodes, link B)Bisection bandwidthNotes
BusB (shared)BOne transfer at a time, but broadcast is free
RingP × B2 × BMultiple simultaneous transfers; messages may hop
2-D mesh / 2-D torus~2P × BO(√P × B)Used in many supercomputers
n-cube (Boolean n-cube)n × P × B / 2O(P/2 × B)2ⁿ nodes, n links per switch, n nearest neighbors
Fully connectedP(P-1)/2 × B(P/2)² × BWonderful but prohibitively expensive

Multistage Networks

  • Crossbar — any node to any other in one pass. n² switches.
  • Omega network — fewer switches (2n log n) but contention possible (can't always route arbitrary permutations). ~25% the switches of a crossbar but limited routing flexibility.

Pretty topologies often don't survive contact with silicon: chip area is 2D, long wires cost more energy and are slower. Practical multicore on-chip networks tend to be simple grids.

8.9 Cluster Networking

Hardware Players

EthernetDominant cluster interconnect. 1 Gb/s standard, 10 Gb/s in datacenters.
NICNetwork Interface Card. Has DMA engine, MAC chips (encode preamble + CRC), PHY chips (physical layer).
PCIePeripheral Component Interconnect Express. Each "lane" = 4 wires (2 RX + 2 TX). PCIe 1.1 = 2 Gb/s/lane; 2.0 = 4 Gb/s; 3.0 = 8 Gb/s.
MAC addressMedia Access Control — 48-bit unique Ethernet ID per NIC.

I/O Mechanisms key

MechanismDefinition
Memory-mapped I/OAddress regions assigned to I/O devices. Reads/writes to those addresses are commands to the device. OS protects via address translation.
DMADirect Memory Access — device controller transfers data to/from memory without involving the processor.
Interrupt-driven I/ODevice asserts interrupt when an I/O operation completes; OS handles. Asynchronous to instruction execution.
PollingProcessor periodically checks status bits — alternative to interrupts. Reduces wakeup latency but uses cycles.

Five-Step NIC Driver (TX and RX)

  1. Driver allocates a packet buffer in host memory; copies user data in (TX) or allocates an empty one (RX).
  2. Driver writes I/O descriptor to NIC register (buffer addr + length).
  3. NIC's DMA copies the packet over PCIe (TX: from buffer to NIC; RX: from NIC into buffer).
  4. DMA interrupts processor when transfer is complete.
  5. Driver de-allocates (TX) or copies into user space (RX).

Performance Optimizations key

  • Zero-copy: DMA reads from / writes to user space directly, skipping intermediate OS buffers.
  • User-space communication: bypass OS to avoid context switch overhead.
  • Polling instead of interrupts: avoid low-power wake latency. Best for low-latency small packets.
  • Hardware preamble + CRC computation: NIC offloads the easy stuff.
  • Direct Data IO (DDIO): Recent Intel — dedicate up to 10% of last-level cache as a scratchpad for DMA. Latency and bandwidth win.

TCP/IP

Ethernet is the foundation; TCP/IP layers reliable communication on top. TCP retransmits until acknowledged; IP includes header checksum for destination integrity.

Roofline Model key

Williams, Waterman, Patterson 2009. Visualization tool that bounds achievable performance from above.

Arithmetic Intensity

Arithmetic Intensity = (Total FP operations) / (Total bytes accessed from main memory)

Higher AI = more compute per byte = less likely to be memory-bound. Some kernels (e.g., dense matrix) have AI that grows with problem size.

Roofline Equation

Achievable GFLOPS = MIN( Peak Compute, Peak Memory Bandwidth × Arithmetic Intensity )

Plot: log(GFLOPS) vs log(FLOPs/byte). Two lines:

  • Horizontal "roof" at peak compute (GFLOPS)
  • Diagonal slope = peak memory bandwidth (GB/s)

Ridge point = where the two meet. If your kernel's AI is to the LEFT of the ridge → memory-bound. To the RIGHT → compute-bound.

Optimization Ceilings

Add progressive ceilings under the roofline showing what optimizations buy you:

  • Computation: (1) FP op mix balance (equal mul/add); (2) Improve ILP and apply SIMD (e.g., AVX 4×64-bit).
  • Memory: (1) Software prefetching; (2) Memory affinity (NUMA-aware data placement).

Stream Benchmark

Used to find the actual achievable peak memory bandwidth (the diagonal line). Raw DRAM bandwidth would be too optimistic.

8.11 Real Stuff: Core i7 960 vs NVIDIA Tesla GTX 280

Lee et al. (Intel) compared a quad-core Core i7 960 (multimedia SIMD) vs Tesla GTX 280.

Specs Snapshot

Core i7 960GTX 280
DP FP peak51.2 GFLOPS78 GFLOPS (1.5×)
SP FP peak102.4 GFLOPS624 GFLOPS (~6×)
Memory bandwidth16.4 GB/s127 GB/s (~7.7× DRAM pin)
DP ridge point3.1 FLOP/byte0.6 FLOP/byte (much easier to reach roof)

14-Kernel Performance Findings

  • Memory-bound kernels (LBM, SAXPY, SpMV): GTX wins by 5–5.3× — explained by 4.4× memory bandwidth.
  • Compute-bound (SGEMM, Conv, FFT, MC, Bilat): GTX wins 1.8–5.7×. Bilat wins 5.7× thanks to GPU hardware transcendentals.
  • Cache-friendly (RC, Search, Sort): Core i7 cache blocking saves it; smaller margins or i7 wins. Sort: i7 1.25× FASTER than GTX (the split SIMD primitive bloats instruction count).
  • Gather-scatter heavy (GJK): GTX wins 15.2× — vector-style gather-scatter is missing from x86 SIMD extensions. 6 of 14 kernels would benefit from gather-scatter on i7.
  • Synchronization (Hist, Solv): Solv: i7 actually wins 2× because GTX 280 lacks the right memory consistency model.

Successor (Fermi) addressed many GTX 280 weaknesses: faster DP FP, faster atomics, real caches.

8.13 Fallacies and Pitfalls core

Fallacy: Amdahl's Law doesn't apply to parallel computers.

No. Amdahl's Law always applies. The 1987 "broken Amdahl" claim was actually weak scaling — the sequential portion was constant and the parallel work grew with the problem.

Fallacy: Peak performance tracks observed performance.

No. Marketing multiplies peak per-CPU by core count, then assumes perfect scaling — Amdahl's Law and the roofline model both refute this.

Pitfall: Not developing software to take advantage of multiprocessor architecture.

Example: SGI OS protected its page table with a single lock — fine on a uniprocessor. On a multiprocessor where many processes initialize many pages at startup, the lock serializes everyone. The fix: many small locks on smaller portions of the page table. The pitfall: even task-level parallelism with no shared data was hurt by this OS-level bottleneck.

Fallacy: You can get good vector performance without providing memory bandwidth.

No. DAXPY uses ~1.5 memory references per FP op. Even if the FP unit took zero time, memory bandwidth caps performance. Cache blocking on the Cray-1 nearly doubled performance simply by reducing memory references per FLOP.

8.14 Concluding Remarks

  • The IT industry has tied its future to parallel computing.
  • SaaS + clusters deliver high-availability services at scale (5-nines).
  • Warehouse-Scale Computers and mobile clients are reshaping server and microprocessor design.
  • SIMD/vector are easier than MIMD and more energy-efficient. SIMD potential growth ≈ 2× MIMD growth over the next decade.
  • Clusters dominate scientific computing — >80% of the 2012 Top500.
  • Open-source movement is critical to the parallel revolution.
  • Just-in-time compilation and autotuning let software adapt to growing core counts.

SIMD Persistence

Pure SIMD multiprocessors (Illiac IV, Connection Machine, MasPar) failed: too inflexible, can't ride microprocessor cost curves. But SIMD lives on as instruction-set extensions (MMX/SSE/AVX) on commodity CPUs and as the core idea behind GPU SIMT.

Common Exam Traps

  • "Amdahl's Law doesn't apply to weak scaling." It does. Weak scaling lets you do MORE work proportionally — it doesn't make the sequential portion vanish. The 1987 "broken Amdahl" claim was just careful weak scaling, not a refutation.
  • "Concurrent software requires parallel hardware." No. Concurrent software (e.g., an OS) can run on a uniprocessor; the OS just multiplexes a single CPU.
  • "SIMD and SPMD are the same thing." No. SIMD = one instruction stream broadcast across many data elements. SPMD = one PROGRAM running on many processors, each free to take different branches at runtime — typical MIMD programming style.
  • "Vector and SIMD extensions are interchangeable." No. Vector encodes length in a register (binary-compatible across widths), supports gather-scatter; SIMD extensions encode length in the opcode (new opcodes per width) and require contiguous aligned access.
  • "Hardware multithreading and multicore are the same thing." No. Multithreading shares a single core's functional units across multiple threads. Multicore replicates entire cores. They compose: a multicore chip can have SMT per core.
  • "NUMA always loses to UMA." No. UMA scales poorly — at large processor counts the shared memory becomes a bottleneck. NUMA scales bigger and offers low local-memory latency.
  • "Clusters are message-passing because they have to be." Largely yes, since each cluster node runs its own OS and has its own physical memory. But the deeper reason is dependability: separate memories make it easy to replace a failed node without bringing the cluster down.
  • "Interrupts are always faster than polling." No. For low-latency small packets in a tight loop, polling avoids interrupt setup, low-power exit, and context switches — faster than interrupts.
  • "GPUs don't have caches." Misleading. They have small streaming caches and texture caches. They simply don't rely on caches for latency hiding (multithreading does that).
  • "Roofline's diagonal line is the raw DRAM bandwidth." No. Use the Stream benchmark's measured bandwidth — raw DRAM is too optimistic to be useful.
  • "Peak performance is what you'll see in real workloads." Almost never. Marketing multiplies peak per chip × number of chips and assumes perfect scaling. Both Amdahl and the roofline model show why this is fantasy.
  • "More threads always equals more performance." No. If the working set fits in L1 already (e.g., 32×32 DGEMM), threading overhead can slow you down. SMT also limits you when threads contend for shared functional units (Sandy Bridge AVX example).
  • "You only need cache coherence in shared memory; clusters are coherent because they have separate memories." Cluster nodes don't share memory at all — there's no coherence problem because there's no shared cache. Coherence is a problem inside an SMP, not across cluster nodes.
  • "Strong scaling is always preferable." Depends. Strong scaling is what you want for real-time problems; weak scaling matches "bigger problems on bigger machines" — common in databases (TPC-C), scientific simulation, and search indexing.
  • "OpenMP is hard to use." No. Adding ONE pragma (#pragma omp parallel for) to a clean loop typically parallelizes it. The hard part is making the underlying algorithm efficient on a parallel machine.
  • "All multimedia SIMD instructions support gather/scatter." No. Most x86 SIMD requires contiguous, 16-byte-aligned access. Haswell added gather (not scatter). Vector ISAs always supported both — that's the historical advantage.
