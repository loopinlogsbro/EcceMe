6.1 Introduction core

Performance is determined by instruction count, clock cycle time, and CPI. The processor implementation determines both clock cycle time and CPI. This chapter builds LEGv8 datapath and control for two implementations: single-cycle and pipelined.

Implemented LEGv8 Instruction Subset

Memory-referenceLDUR, STUR — load/store using base register + 9-bit signed offset
Arithmetic-logicalADD, SUB, AND, ORR — R-format, three register operands
BranchCBZ (compare & branch zero, 19-bit offset), B (unconditional, 26-bit offset)

Common Steps for Every Instruction

  • Step 1: Send PC to instruction memory; fetch instruction.
  • Step 2: Read one or two registers using instruction fields.
  • Step 3+: Diverges by instruction class (ALU op, memory access, branch).

All instruction classes except unconditional branch use the ALU after reading registers.

6.2 Logic Design Conventions core

Two types of logic elements and edge-triggered clocking are fundamental to the LEGv8 datapath design.

Combinational elementOutput depends only on current inputs. No internal storage. Examples: ALU, adder, AND gate. Given the same inputs, always produces the same output.
State element (sequential)Has internal storage; output depends on inputs AND stored state. Examples: register file, PC, instruction memory, data memory. Requires a clock input.

Clocking

Clocking methodologyDetermines when data is valid and stable relative to the clock — makes hardware predictable.
Edge-triggered clockingAll state changes occur on a clock edge (rising edge in this chapter). Allows reading and writing the same register in one cycle without a race condition.
Asserted / DeassertedAsserted = logically high (1). Deasserted = logically low (0). Used instead of "high/low" to avoid confusion with voltage polarity.
Clock cycle must be long enough for signals to propagate through all combinational logic between state elements. State elements are written on the rising clock edge; reads can occur at any time.

A state element that is not updated on every clock edge requires an explicit write control signal. The PC has no write control because it is written every cycle.

6.3 Building a Datapath core

Build the datapath bottom-up by identifying what each instruction class needs, then combine with multiplexors where paths share hardware.

Program counter (PC)64-bit register holding address of next instruction. Written every cycle — no write control needed.
Register file32 general-purpose 64-bit registers. Two read ports (5-bit register number in, 64-bit data out), one write port (RegWrite control). Reads anytime; writes on clock edge.
Sign-extend unitExtends 9-bit offset (LDUR/STUR) or 19-bit offset (CBZ) to 64 bits. Bit 26 of opcode selects which field: 0=9-bit, 1=19-bit.
Data memoryState element. Separate from instruction memory. Has MemRead and MemWrite controls. Reading an invalid address can cause problems.
Datapath elementAny unit that operates on or holds data: instruction memory, data memory, register file, ALU, adders.
Branch target addressPC + (sign-extended 19-bit offset << 2). Shift left 2 = concatenate 00 to low end. Branch taken if CBZ register = 0.

Instruction ClassComponents NeededMux Needed?

Single-cycle design: no datapath resource may be used more than once per instruction, so instruction memory and data memory must be separate.

Instruction Execution Flow (Steps)

R-type (e.g., ADD X9, X20, X21)
  • Fetch instruction, PC+4
  • Read X20, X21 from register file; decode opcode
  • ALU performs operation
  • Write result to X9

  • Read X2; sign-extend offset
  • ALU computes X2 + offset (address)
  • Read data memory at that address
  • Write memory data to X1

Why Single-Cycle is Not Used Today

Clock cycle must accommodate the slowest instruction (LDUR = 800 ps). Even fast instructions waste time. Violates "make the common case fast." Single-cycle CPI = 1 but clock is too long.

Single-cycle timing (example): Instruction memory: 200 ps Register file read/write: 100 ps ALU operation: 200 ps Data memory: 200 ps LDUR total: 200+100+200+200+100 = 800 ps ← sets clock cycle ADD total: 200+100+200+0+100 = 600 ps ← wastes 200 ps

6.5 An Overview of Pipelining core

Pipelining overlaps multiple instructions in execution — like an assembly line. It improves throughput, not individual instruction latency.

PipeliningOverlapping instruction execution. Increases instruction throughput, not latency. Nearly universal in modern processors.
Latency (pipeline)Number of stages between two instructions, or the time to complete one instruction. Pipelining does NOT reduce latency.
Structural hazardHardware cannot support the combination of instructions trying to execute simultaneously (e.g., two instructions need same memory).
Data hazardInstruction cannot execute in the proper cycle because data it needs is not yet available (produced by a prior in-flight instruction).
Control hazardProper instruction cannot execute because the pipeline doesn't yet know which instruction to fetch (branches).
Forwarding (bypassing)Retrieve missing data from internal pipeline buffers rather than waiting for it to be written back to the register file.
Load-use data hazardSpecific hazard: data loaded by LDUR not yet available when the very next instruction needs it. Forwarding alone cannot resolve — requires a stall.
Pipeline stall (bubble)A "do nothing" cycle inserted to resolve a hazard. Stalls freeze the IF and ID stages; a nop propagates into EX.
Branch predictionGuess the outcome of a branch and proceed; fix up if wrong. Reduces control hazard penalty.

Pipelining Performance Example

Three LDUR instructions (each 800 ps single-cycle, 200 ps/stage pipelined): non-pipelined takes 2400 ps; pipelined takes 1400 ps (first complete at 1000 ps, then 200 ps each). For large N, speedup approaches 4x.

STUR  X5, [X0,#32]

6.6 Pipelined Datapath & Control core

Add pipeline registers between each stage to hold intermediate values across clock cycles. Control signals are generated in the ID stage and propagated through pipeline registers to the stage where they are needed.

Pipeline Registers

IF/IDHolds fetched instruction and PC+4 between Instruction Fetch and Decode stages.
ID/EXHolds decoded values, register data, sign-extended immediate, control signals, and register numbers (Rn, Rm, Rd).
EX/MEMHolds ALU result, Zero flag, register Rt/Rd number, write data (for stores), and remaining control signals.
MEM/WBHolds data memory read result, ALU result, register Rd number, and WB control signals.

Control signals generated in ID: WB signals (RegWrite, MemtoReg) travel all the way to WB stage. MEM signals (MemRead, MemWrite, Branch) stop at MEM stage. EX signals (ALUOp, ALUSrc, Reg2Loc) used in EX stage.

Register File Write-Half / Read-Half Trick

Writes occur in the first half of WB; reads occur in the second half of ID. This means an instruction in WB and an instruction in ID can access the same register without a hazard — the read gets the value being written.

6.8 Control Hazards core

Branch outcome is not known until late in the pipeline. Without optimization, 3 instructions are fetched after a branch before we know the target. By moving branch evaluation to ID stage, penalty reduces to 1 bubble.

FlushDiscard instructions in a pipeline by zeroing control signals — converts fetched instructions into nops. Used for taken branches and exceptions.
Branch prediction bufferAlso called branch history table. Small memory indexed by branch instruction address bits, holding 1+ bits indicating recent taken/not-taken.
Dynamic branch predictionPredict branches at runtime using history. Adjusts predictions based on recent behavior of each specific branch.
Correlating predictorUses both local branch behavior AND global recent branch history for higher accuracy with the same number of bits.
Tournament branch predictorMultiple predictors per branch; a selector tracks which has been most accurate and chooses between them.
Branch target bufferCaches the destination PC for a branch instruction. Allows fetching from target before branch is fully evaluated.

Branch Prediction Strategies

StrategyMechanismBest When
Assume not takenContinue sequential fetch; flush if takenBranch taken < 50%
Assume takenFetch from predicted target immediatelyBranch taken > 50%
1-bit dynamicRemember last outcome; predict same next timeHighly biased branches; can mispredict twice on loop exit
2-bit dynamicCounter must be wrong twice before changing predictionStrongly biased branches; mispredicts once on loop exit
2-bit predictor states: Strongly Taken → Weakly Taken → Weakly Not Taken → Strongly Not Taken A prediction changes only after two consecutive mispredictions.

Reducing Branch Penalty

Move branch evaluation from MEM stage to ID stage by: (1) moving branch adder to ID stage (compute branch target early), and (2) testing for zero during ID. This reduces flush from 3 instructions to 1 instruction on a taken branch. Adding IF.Flush control signal zeros the IF/ID register to convert the one fetched instruction into a nop.

6.9 Exceptions core

ExceptionUnscheduled event that disrupts program execution. Internally generated (undefined instruction, overflow, hardware malfunction). LEGv8 uses this term for both internal and external events.
InterruptException caused by an external device (I/O). ARM distinguishes this from internally-generated exceptions.
ELR (Exception Link Register)64-bit register storing the address of the instruction that caused the exception, so the OS can restart or report it.
ESR (Exception Syndrome Register)32-bit register recording the cause of the exception. Values: 8=undefined instruction, 10=arithmetic overflow/underflow, 12=hardware malfunction.
Vectored interruptException address is determined by the cause (different address for each type). LEGv8 uses a single entry point instead.
Precise interruptAlways associated with the exact instruction that caused it. All prior instructions complete; all following are flushed. LEGv8 and most modern processors use this.
Imprecise interruptNot precisely linked to the offending instruction. Easier to implement but harder for the OS to handle. Rare in modern systems.
LEGv8 exception entry address: 0x0000_0000_1C09_0000 On exception: save offending instruction address in ELR record cause in ESR jump to exception entry address flush instructions after the offending instruction

Exception Handling in the Pipeline

Exceptions are treated as control hazards. When an exception is detected (typically in the EX stage): EX.Flush zeros control signals to prevent the instruction from completing. The offending instruction's PC is saved in ELR. PC is loaded with the exception entry address. Instructions fetched after the offender are flushed. Prior instructions complete normally.

Multiple simultaneous exceptions: hardware prioritizes the earliest instruction's exception. The ELR captures that instruction's address.

6.10 Parallelism via Instructions (ILP) core

ILP (Instruction-level parallelism)Parallelism among instructions in a sequential stream. Exploited by pipelining and multiple issue.
Multiple issueLaunch more than one instruction per clock cycle. Enables CPI < 1 (or IPC > 1).
Issue slotsPositions from which instructions can issue in a clock cycle.
Static multiple issue (VLIW)Compiler packages instructions into issue slots at compile time. Very Long Instruction Word — one wide instruction with multiple operation fields.
Dynamic multiple issue (Superscalar)Hardware selects instructions to issue at runtime. Correct execution guaranteed regardless of issue rate. Code is portable across implementations.
SpeculationGuess an instruction's outcome (branch, load address) to begin executing dependent instructions earlier. Must include a mechanism to recover if wrong.
Loop unrollingReplicate loop body multiple times to expose more ILP and reduce loop overhead. Requires register renaming to avoid false dependences.
Register renamingRename architectural registers to a larger set to eliminate antidependences (name dependences) — ordering caused by reuse of register names, not real data flow.
Antidependence (name dependence)Ordering forced by reuse of a register name, not a true data dependence. Register renaming removes it.
Reorder bufferBuffer holding results in a dynamically scheduled processor until safe to commit. Enables in-order commit while allowing out-of-order execution.
Reservation stationBuffer within a functional unit holding operands and the operation. Instruction issues when all operands are available and unit is free.
Out-of-order executionInstructions execute in a different order than fetched, based on data availability. In-order commit still applies.

Two-Issue LEGv8 Example

Instructions paired: ALU/branch slot + load/store slot. Both issue simultaneously. Use latency increases: loads now have 2-instruction use latency. Loop unrolling (4x) + register renaming achieves IPC of 1.88 vs. theoretical max 2.0.

Cortex-A53 Pipeline Structure

Three sections: Instruction Fetch (3 stages, feeds 13-entry instruction queue), Instruction Decode (3 stages), Execute (3 stages + 2 more for FP/SIMD). Fetch retrieves 2 instructions at a time. Stalls: 60% from pipeline hazards, 40% from memory hierarchy.

Intel Core i7 — 8 Steps

x86 instructions are translated into ARM-like micro-operations (micro-ops) before execution. Key stages: fetch 16 bytes → predecode into individual x86 instructions (variable 1–15 bytes) → decode into micro-ops (3 standard decoders + microcode engine) → micro-op buffer with loop stream detection → register renaming + reorder buffer allocation → dispatch to 36-entry reservation stations → 6 functional units execute → commit in-order to register file/memory.

Intel Core i7 — Register Renaming

16 x86 architectural registers are renamed to a larger set of physical registers. The processor maintains a map from architectural to physical. On misprediction, undo mappings back to the last correct instruction.

Self-Test (Likely Exam-Style Q&A)

Logic Design Fundamentals
What is the difference between a combinational element and a state element?
Combinational: output depends only on current inputs, no internal storage (e.g., ALU, adder). State element: has internal storage; output depends on inputs AND stored state; requires a clock signal (e.g., register file, PC, memory).
What is edge-triggered clocking and why is it important?
All state element writes occur on a clock edge (rising edge). This allows reading and writing the same register in one clock cycle without creating a data race or indeterminate values.
What does "asserted" mean for a control signal?
Asserted means the signal is logically high (1). Deasserted means logically low (0). These terms are used because the actual voltage polarity may vary by implementation.
Does the PC need a write control signal? Why or why not?
No. The PC is written on every clock cycle, so no explicit write control is needed. Elements that are not updated every cycle require an explicit write control.
What determines the length of the clock cycle in a synchronous digital system?
The time required for signals to propagate through the longest combinational logic path between two state elements. All signals must be stable before the next clock edge.
Datapath Construction
Why does a single-cycle LEGv8 implementation require separate instruction memory and data memory?
No datapath resource can be used more than once per instruction in a single-cycle design. Both LDUR and instruction fetch require memory access in the same clock cycle.
How wide are the register number inputs to the register file? How wide are the data outputs?
Register number inputs are 5 bits (to address one of 32 registers). Data inputs and outputs are 64 bits (LEGv8 is 64-bit).
What does the sign-extend unit do for LDUR vs. CBZ?
For LDUR/STUR: sign-extends the 9-bit offset field (instruction bits 20:12) to 64 bits. For CBZ: sign-extends the 19-bit offset (instruction bits 23:5) to 64 bits. Opcode bit 26 selects which: 0=9-bit, 1=19-bit.
How is the branch target address computed for CBZ?
Branch target = PC + (sign-extended 19-bit offset × 4). The ×4 is achieved by shifting left 2 (concatenating 00 to the low end of the sign-extended value). No actual hardware shifter is needed — just rewiring.
How many multiplexors are needed in the combined single-cycle datapath, and what do they select?
At minimum three: (1) ALUSrc — selects register vs. sign-extended immediate for ALU input 2. (2) MemtoReg — selects ALU result vs. memory data for register write-back. (3) PCSrc — selects PC+4 vs. branch target for next PC. UncondBranch adds a fourth (or an OR gate).
Control Unit and ALU Control
What are the ALUOp values and what do they mean?
00 = add (LDUR/STUR address calculation). 01 = pass input b (CBZ zero test). 10 = R-type: consult opcode bits to determine operation (ADD, SUB, AND, ORR).
What ALU control value is used for a STUR instruction?
0010 (add). STUR uses ALUOp=00, which always maps to add, regardless of opcode. It needs to compute the memory address = base register + offset.
What is a don't-care term in a truth table?
An X input: the output does not depend on that input's value for that row. For example, when ALUOp=00, the opcode bits are don't-cares because we always add regardless.
How is the PCSrc signal generated? Why is it not directly from the control unit?
PCSrc = Branch AND Zero. The control unit asserts Branch for CBZ instructions; the ALU Zero output indicates the register was 0. Only if both are true should we branch. This is a derived signal — the control unit doesn't know the register value at decode time.
For a load instruction (LDUR), what are the values of ALUSrc, MemRead, MemtoReg, RegWrite?
ALUSrc=1 (immediate offset), MemRead=1, MemtoReg=1 (memory data goes to register), RegWrite=1.
For a store instruction (STUR), what are the values of MemWrite, RegWrite, MemRead?
MemWrite=1, RegWrite=0 (no register is written), MemRead=0.
Pipelining Concepts and Performance
Does pipelining reduce instruction latency? Does it improve throughput?
No, pipelining does NOT reduce latency. A 5-stage pipeline still takes 5 clock cycles per instruction. Pipelining improves throughput by overlapping multiple instructions.
What sets the clock cycle time in a pipelined processor?
The slowest pipeline stage. In the LEGv8 example, the slowest stage is 200 ps (memory access or ALU), so the clock cycle is 200 ps even though register file access is only 100 ps.
What is the ideal speedup from pipelining, and when is it achieved?
Ideal speedup = number of stages (5x for 5 stages). This is approached when all stages take equal time and the number of instructions is much larger than the number of stages (pipeline is full most of the time).
What does the laundry analogy teach about pipelining?
Pipelining doesn't make a single load of laundry finish faster — it makes many loads finish faster by keeping all resources busy simultaneously. Throughput improves; latency does not.
What three properties of the LEGv8 ISA make pipelining easier?
(1) Fixed-length instructions (32 bits) — easy to fetch and decode. (2) Regular format — source and destination register fields at consistent bit positions. (3) Memory operands only in loads/stores — address calculation and memory access can be cleanly separated into EX and MEM stages.
What registers are written during the first half of WB? What happens in the second half of ID?
Register writes occur in the first half of WB; register reads occur in the second half of ID. This means an instruction in WB writing a register and an instruction in ID reading the same register in the same clock cycle is safe — the read gets the newly written value.
Data Hazards and Forwarding
What are the two hazard conditions that trigger EX-stage forwarding (ForwardA/B = 10)?
EX/MEM.RegWrite is asserted AND EX/MEM.RegisterRd ≠ 31 (not XZR) AND EX/MEM.RegisterRd matches ID/EX.RegisterRn1 (or Rm2 for ForwardB).
When is MEM-stage forwarding used instead of EX-stage forwarding?
When the hazard is two instructions back (MEM/WB has the value, not EX/MEM). Same conditions but using MEM/WB.RegisterRd. If both EX and MEM hazards exist for the same register, EX takes priority (more recent).
Why can't forwarding alone resolve a load-use hazard?
The loaded data from LDUR is not available until the end of the MEM stage (clock cycle 4 for LDUR), but the instruction immediately after needs it at the start of EX (also clock cycle 4 for that instruction). The data would need to go backward in time. One stall cycle is required.
What happens in the pipeline when a load-use stall is inserted?
The PC and IF/ID pipeline register are frozen (instruction in IF is re-fetched with the same PC). The ID/EX control signals are zeroed to create a nop (bubble). After one stall cycle, forwarding from MEM/WB handles the hazard.
Does forwarding help STUR and CBZ instructions in addition to R-type instructions?
Yes. STUR uses Rt (bits 4:0) for the data to store, and CBZ tests a register. The pipeline uses the Reg2Loc mux output (which handles the Rt field) in place of Rm when detecting hazards, so forwarding is correctly applied to these instructions too.
Given: ADD X1,X1,X2 / ADD X1,X1,X3 / ADD X1,X1,X4 — what forwarding is used?
For the second ADD: EX hazard, forward X1 from EX/MEM. For the third ADD: EX hazard from second ADD (EX/MEM), which takes priority over MEM hazard from first ADD (MEM/WB). Most recent result wins.
Control Hazards and Branch Prediction
In the unoptimized pipeline, how many instructions are fetched after a branch before the outcome is known?
3 instructions. Branch outcome is determined in MEM (clock cycle 4). So instructions at PC+4, PC+8, PC+12 are fetched before we know whether to branch.
After moving branch evaluation to the ID stage, how many instructions must be flushed when a branch is taken?
Only 1 — the instruction fetched at PC+4 while the branch was in ID. It is converted to a nop via the IF.Flush signal that zeros the IF/ID pipeline register.
What is the performance impact of always stalling on branches (for a 5-stage pipeline)?
Branches are ~17% of instructions (SPECint). Stalling one cycle per branch yields CPI = 1 + 0.17×1 = 1.17, a 17% slowdown. Longer pipelines incur a larger penalty.
What is the weakness of a 1-bit branch predictor for a loop that iterates 9 times?
It mispredicts twice per loop execution: once on the first iteration (prediction was flipped by the previous exit) and once on the last iteration (predicts taken, but branch exits). 80% accuracy despite 90% taken rate.
How does a 2-bit predictor fix the 1-bit predictor's weakness?
A prediction must be wrong twice before it changes. A loop that exits occasionally mispredicts only once (on exit), not twice. Best-case accuracy for a 90%-taken branch goes to 90%.
What is a correlating predictor?
Uses both local branch history (behavior of this specific branch) and global history (recent behavior of all branches) to choose a prediction. More accurate than pure local or global alone for the same number of bits.
Exceptions
What registers are used to handle exceptions in LEGv8?
ELR (Exception Link Register): saves address of the offending instruction. ESR (Exception Syndrome Register): records cause code (8=undefined instruction, 10=arithmetic overflow/underflow, 12=hardware malfunction).
What is the LEGv8 exception entry address?
0x0000_0000_1C09_0000. All exceptions jump to this single entry point; the OS reads ESR to determine the cause.
What is the difference between LEGv8's approach and vectored interrupts?
LEGv8 uses a single entry point for all exceptions; the OS reads ESR to distinguish causes. Vectored interrupts use different addresses for different exception types, so the OS knows the cause from the address itself.
What is a precise interrupt?
An interrupt precisely associated with the instruction that caused it. All prior instructions complete normally; all following instructions are flushed. LEGv8 and nearly all modern processors support precise interrupts.
If an exception is detected in the EX stage for instruction ADD X1,X2,X1, which instructions continue and which are flushed?
Instructions before ADD in the pipeline (AND, ORR if present) complete normally. ADD and all instructions fetched after it are flushed. The address of ADD's successor is saved in ELR, not ADD itself — the OS can restart ADD from there.
When multiple exceptions occur simultaneously, which is handled first?
The earliest instruction's exception takes priority. Hardware sorts exceptions so the exception associated with the earliest (oldest in program order) instruction is serviced first.
ILP, Multiple Issue, and Superscalar
What is the maximum theoretical CPI for a 4-way multiple-issue processor?
0.25 (CPI = 1/IPC = 1/4). In practice, data/control dependences and memory stalls prevent reaching this maximum.
What is VLIW and what is its main advantage/disadvantage?
Very Long Instruction Word — static multiple issue where the compiler packs operations into fixed-format wide instructions. Advantage: simple hardware. Disadvantage: code is not portable across different pipeline implementations; recompilation needed.
Why does a superscalar processor use dynamic scheduling even though the compiler can also schedule code?
Three reasons: (1) cache misses create unpredictable stalls the compiler can't see. (2) Dynamic branch prediction creates runtime instruction order the compiler can't know. (3) Pipeline structure changes between implementations; hardware adaptation is transparent to software.
What is the purpose of the reorder buffer in a dynamically scheduled pipeline?
Holds results until they can be committed in program order (in-order commit). Enables out-of-order execution while maintaining precise interrupt support and correct exception handling.
What is register renaming, and what problem does it solve?
Renaming architectural registers to a larger physical register set. Solves antidependences (name dependences) — cases where two independent instruction sequences happen to use the same register name but share no actual data. Allows them to execute in parallel.
What is loop unrolling and what is its cost?
Replicating the loop body multiple times to expose more ILP. Cost: larger code size and more registers used. Benefit: reduced loop overhead and more opportunities for parallel issue.
What is in-order commit and why is it required?
Results are committed (written to register file/memory) in program fetch order, even if executed out of order. Required to support precise interrupts — if an exception occurs, all prior instructions have committed and none after have.
ARM Cortex-A53 and Intel Core i7
How many pipeline stages does the ARM Cortex-A53 have?
8 stages total: 3 fetch stages + 3 decode stages + 3 execute stages (with 2 additional for FP/SIMD). The AGU (Address Generation Unit) manages branch prediction.
What is the ARM Cortex-A53's branch misprediction penalty?
8 clock cycles. The misprediction empties the pipeline because the instruction queue cannot decouple fetch from execute during a misprediction.
Why does the Intel Core i7 translate x86 instructions to micro-ops?
x86 instructions have variable length (1–15 bytes) and complex semantics. Translating to fixed, simple micro-ops allows using a clean, efficient dynamically scheduled pipeline. The pipeline operates on micro-ops, not native x86 instructions.
What is the Intel Core i7's ideal CPI and actual median CPI for SPEC2006?
Ideal CPI = 0.25 (4-way issue in theory). Actual median CPI = 0.79. Best case = 0.44, worst case = 2.67.
What fraction of the ARM Cortex-A53's stalls come from pipeline hazards vs. memory hierarchy?
60% from pipeline hazards (branch mispredictions, structural hazards, data dependences between paired instructions). 40% from memory hierarchy stalls.

Common Exam Traps

  • "Pipelining reduces the time it takes a single instruction to execute." No. Pipelining reduces the average time between instructions (throughput), not the latency. A 5-stage pipeline still takes 5 clock cycles per instruction.
  • "The single-cycle implementation has a CPI of 1, so it must be fast." No. CPI=1 but the clock cycle is set by the slowest instruction (800 ps for LDUR). Pipelining uses 200 ps cycles with CPI≈1 for better throughput.
  • "Forwarding can resolve any data hazard." No. The load-use hazard (LDUR immediately followed by an instruction using the loaded value) cannot be resolved by forwarding alone — one stall cycle is required.
  • "The PC needs a write control signal." No. The PC is written every clock cycle, so no explicit write control is needed. Only state elements that are NOT written every cycle need a write control signal.
  • "The register file needs a read control signal." No. The register file always outputs the contents of whatever register numbers are on the read inputs. Reads are combinational — no control needed. Only writes are controlled (RegWrite).
  • "A 1-bit branch predictor achieves 90% accuracy for a 90%-taken branch." No. It achieves only 80% — it mispredicts twice per loop iteration (once on first, once on last). A 2-bit predictor achieves 90%.
  • "STUR reads from memory." No. STUR writes to memory. MemRead=0, MemWrite=1. The register value (Rt) is read from the register file and stored into memory.
  • "For CBZ, the ALU performs a subtraction to check the zero condition." No. CBZ uses ALU control 0111 (pass input b), which passes the register value directly. The Zero output detects if the passed value is zero.
  • "Register renaming renames physical registers to architectural registers." No. It renames architectural registers (the ISA-visible ones) to a larger set of physical registers. The direction is architectural → physical.
  • "Dynamic scheduling (superscalar) commits instructions out of order." No. Execution is out-of-order, but commit is always in-order. This is required for precise interrupt support.
  • "All exceptions are detected in the same pipeline stage." No. Different exceptions are detected at different stages. Arithmetic exceptions (overflow) are detected in EX; undefined instructions may be detected in ID; hardware malfunctions can be EX or later.
  • "The instruction address saved in ELR is the address of the exception handler." No. ELR saves the address of the offending instruction (so the OS can know where to restart or report). The exception handler address is the fixed entry point 0x0000_0000_1C09_0000.
  • "VLIW code is portable across different VLIW processors." No (usually). Code scheduled for one VLIW pipeline may need recompilation for a different pipeline depth or issue width — this is a major disadvantage of VLIW vs. superscalar.
  • "A structural hazard is the same as a data hazard." No. Structural: hardware conflict (two instructions need same resource). Data: an instruction needs a value not yet produced. Control: next instruction unknown (branches). Three distinct types.
