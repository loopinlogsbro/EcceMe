10.1 Introduction core

Core Definitions

GPUGraphics Processing Unit. A processor optimized for 2D/3D graphics, video, visual computing, and display. Modern GPU = highly parallel, highly multithreaded multiprocessor.
Visual computingMix of graphics processing + computing that lets you visually interact with computed objects via graphics, images, and video.
Heterogeneous systemA system combining different processor types — e.g., a PC's CPU + GPU.
APIApplication Programming Interface. OpenGL (open standard) and Direct3D (Microsoft) are the two main graphics APIs.

VGA → GPU Evolution Timeline

  • 1990: No GPU — graphics done by VGA controller (memory controller + display generator over DRAM).
  • 1997: VGA controllers add 3D acceleration (triangle setup, rasterization, texture mapping, shading).
  • 2000: Term "GPU" coined when single chips fold in the whole graphics pipeline.
  • Programmability progresses: indexed → integer/fixed → single-precision FP → double-precision FP.
  • Recent: hundreds of cores, thousands of threads, programmable in C/C++ via CUDA.

GPGPU vs GPU Computing vs CUDA key

TermMeaning
GPGPUGeneral-Purpose computing on GPU using a graphics API and graphics pipeline (awkward — you cast your problem as a graphics op).
GPU computingUsing the GPU for computing via a parallel programming language and compute API — bypasses the graphics pipeline.
CUDACompute Unified Device Architecture. NVIDIA's scalable parallel programming model and software platform built on C/C++. SPMD style.

10.2 GPU System Architectures core

Heterogeneous CPU-GPU System (Today)

  • Discrete GPU + CPU with separate memories. GPU attached via PCI-Express (16-lane PCIe 2.0 = 16 GB/s peak, 8 GB/s each direction).
  • GPU and CPU can access each other's memory (lower bandwidth than direct).
  • UMA (Unified Memory Architecture): no dedicated GPU memory — uses CPU system memory only. Lower performance — limited by system memory bandwidth and latency.
  • Multi-GPU systems: e.g., NVIDIA SLI (Scalable Link Interconnect), 2–4 GPUs working in parallel.
  • Integrated GPUs sit on the north bridge (Intel) or chipset (AMD) — with or without dedicated graphics memory.

Memory Visibility

GPU has its own MMU. Page tables managed by the OS kernel. Accesses can be coherent or non-coherent PCI-Express transactions, set by an attribute in the GPU page table. CPU reads GPU memory via an aperture in PCIe address space.

PCI-Express & AGP

PCI-Express (PCIe)Standard system I/O interconnect using point-to-point links. Configurable lane count and bandwidth.
AGPOlder PCI extension dedicated to graphics — provided up to 8× PCI bandwidth to a single card slot.

Logical Graphics Pipeline (Direct3D 10 / OpenGL)

Programmable stages (blue) interleaved with fixed-function stages (white):

  1. Input assembler — collect vertices into primitives (points, lines, triangles).
  2. Vertex shader (programmable) — per-vertex; transform position, compute lighting/color.
  3. Geometry shader (programmable) — per-primitive; can add or drop primitives.
  4. Setup / rasterizer — generate pixel fragments covered by primitives.
  5. Pixel (fragment) shader (programmable) — per-fragment; interpolation, texturing, coloring.
  6. Raster operations / output merger — Z-buffer test, stencil test, blend with framebuffer.

Basic Unified GPU Architecture key

  • Earlier GPUs had separate processors per shader stage (vertex, geometry, pixel). Modern unified GPUs run all three on the same array of SP cores.
  • Example (Tesla / GeForce 8800 baseline): 112 SP cores organized as 14 streaming multiprocessors (SMs) — 8 SP cores per SM.
  • Each SM contains: 8 SP cores, 2 SFUs, multithreaded instruction unit, instruction cache, constant cache, shared memory.
  • SMs connect through an interconnection network to four 64-bit DRAM partitions.
  • Two SMs share a texture unit and a small streaming texture L1 cache (filter regions overlap → caching pays off).
  • Unification benefits: dramatic scalability (more cores = more throughput), automatic load balancing across stages, the same array can serve graphics or compute work.

10.3 Programming GPUs & CUDA core

GPUs offer 2–3 orders of magnitude more thread/data parallelism than CPUs. Programmer writes code for one thread; GPU runs millions of instances in parallel. The same program scales transparently across hardware sizes.

Graphics Programming Languages

HLSLHigh-Level Shading Language (Microsoft, used with Direct3D).
Cg"C for graphics" (NVIDIA). Both have C-like syntax. Lack general memory access, pointers, file I/O, recursion. I/O is implicit through the graphics pipeline.
ShaderA program that operates on graphics data — a vertex, primitive, or pixel fragment.
Texture1D/2D/3D array supporting filtered, interpolated lookups.

CUDA Hierarchy key

LevelDefinitionMemory
ThreadOne instance of a kernel functionLocal memory (private, in DRAM)
Thread block (CTA)Set of concurrent threads (up to 512); can cooperate via barrier sync and shared memoryShared memory (per-block, on-chip SRAM)
GridSet of thread blocks executing the same kernelGlobal memory (per-application, in DRAM)

Key CUDA Vocabulary

KernelA function written for a single thread, designed to be executed by many threads in parallel.
Thread block / CTACooperative Thread Array. Implements a CUDA thread block as one or more warps of parallel threads.
GridSet of independent thread blocks executing the same kernel.
Synchronization barrierThreads block at the barrier until all threads in the block arrive. Implemented via __syncthreads() in CUDA / bar.sync in PTX.
Atomic memory opRead-modify-write that completes without intervening access. CUDA atom.op.u32: add, min, max, and, or, xor, exchange, cas.
SPMDSingle-Program Multiple Data. All threads execute the same program; CUDA is a flexible SPMD model.

Kernel Launch Syntax

kernel<<<dimGrid, dimBlock>>>(args);
__global__ void saxpy(int n, float a, float *x, float *y) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) y[i] = a * x[i] + y[i];
}
saxpy<<<(n+255)/256, 256>>>(n, 2.0f, x, y);

Each thread gets a threadIdx within its block and a blockIdx within the grid. Both have .x .y .z dimension fields.

Three CUDA Memory Spaces

SpaceVisibilityStorageDeclaration
LocalPer-threadExternal DRAMAutomatic for spilled registers and locals
SharedPer-block (CTA)On-chip SRAM, low latency__shared__
GlobalAll threads, persistent across kernelsExternal DRAM__device__ or cudaMalloc

CUDA Restrictions

  • Threads/thread blocks can only be created by launching a kernel — not from within one.
  • Thread blocks must execute independently; no inter-block communication during a kernel (must launch a new kernel to combine results, or use atomics on global memory).
  • No recursion in kernels (would need stack space for tens of thousands of threads).
  • Programmer must cudaMemcpy to/from device memory (CPU-GPU DMA over PCIe).

10.4 Multithreaded Multiprocessor Architecture core

A GPU is a multiprocessor of multiprocessors: many SMs, each highly multithreaded. Example SM: 8 SP cores tightly coupled, executing up to 512 threads (Tesla SM described later: up to 768).

Why Massive Multithreading?

  • Cover the latency of memory loads and texture fetches from DRAM (hundreds of cycles).
  • Support fine-grained parallel programming models.
  • Virtualize physical processors as threads/blocks for transparent scaling.
  • Simplify the programming model down to writing a serial program for one thread.

While one thread waits on memory, the SP runs another thread. Hardware multithreading = zero scheduling overhead.

SP Core Resources

  • Scalar integer + FP arithmetic. Pipelined; one scalar instruction per thread per clock at 1.2–1.6 GHz.
  • Register file: 1024 general-purpose 32-bit registers, partitioned among threads. Programs declare register demand — compiler trades thread count vs spill cost.
  • SP core hardware-multithreaded for up to 64 threads. Pixel shaders typically use ≤16 regs (so 64 threads/SP); CUDA kernels often use 32 regs (so 32 threads/SP).

SIMT (Single-Instruction Multiple-Thread) key

ConceptDefinition
SIMTOne instruction applied to multiple independent threads in parallel — not just multiple data lanes.
WarpThe set of parallel threads that execute the same instruction together. Tesla warp size = 32 threads.
IssueEach warp instruction broadcasts synchronously to its 32 active threads. On the example multiprocessor, executes as 4 sets of 8 threads over 4 clocks.
DivergenceIf threads in a warp branch differently, execution serializes for each path; warp re-converges when all paths complete. A perfectly split if-else is 50% efficient. Different warps execute independently — no penalty for divergence between warps.
SchedulerSelects a ready warp every issue cycle (round-robin + priority + scoreboard for instruction dependencies). For 8-warp round-robin, the same warp re-issues every 32 cycles → can hide instruction latency up to 32 cycles.

SIMT vs SIMD

SIMDOne instruction → multiple data lanes. Programmer must express parallelism explicitly in vector instructions; lane alignment is required for memory access.
SIMTOne instruction → multiple independent threads. Each thread has its own scalar registers, PC, branch path. Hardware finds data parallelism among threads at runtime. Coalesced memory access is preferred but not required.

SIMT thread programs look like serial code. Compare to SIMD where you must hand-code vector ops with edge-case handling.

PTX (Parallel Thread eXecution) ISA

  • Stable virtual ISA that compilers target. Translated/optimized at install/run time to binary microinstructions for the specific GPU.
  • Register-based load/store scalar ISA.
  • Format: opcode.type d, a, b, c; — .type is .b/.u/.s/.f with width 8/16/32/64 bits.
  • Predication: prefix instruction with @p or @!p.
  • Scalar instructions (unlike older 4-component vector GPU ISAs); compiler-friendly. Texture instructions stay vector-based.
  • Memory ops: ld.global / st.global / ld.shared / st.shared / ld.local / st.local / atom.op.u32.
  • Sync: bar.sync.

Special Function Unit (SFU)

  • 2 SFUs per multiprocessor. Computes 1 special-function result per cycle → roughly ¼ the simple-instruction rate of the 8 SPs.
  • Functions: reciprocal, reciprocal sqrt, log₂, 2^x, sin/cos — accuracy 22–24 mantissa bits.
  • Implementation: quadratic interpolation based on enhanced minimax approximations.
  • Also performs FP planar attribute interpolation for pixel shaders.
  • Can also execute mul.f32 concurrently with the SPs → up to +50% peak compute rate for the right instruction mix.

10.5 Parallel Memory System core

The memory system is the most important determiner of GPU performance. GeForce 8800 example: 32 pixels per clock × 600 MHz × ~28 bytes/pixel ≈ 896 bytes per clock of memory bandwidth demand.

GPU Memory Subsystem Characteristics

  • Wide — many pins, many DRAM chips for large total bus width.
  • Fast — aggressive signaling for high bits/sec per pin (DDR, GDDR).
  • Optimized for throughput, not latency — these are fundamentally in conflict; GPUs choose throughput.
  • Compression (lossy and lossless).
  • Caches and request-coalescing structures to reduce off-chip traffic.

DRAM Reality

  • DRAMs internally arranged as banks (4–8) × rows (~16,384) × bits per row (~8192).
  • Activating a row takes dozens of cycles; once activated, columns randomly accessible every ~4 clocks.
  • DDR transfers on both clock edges; 1 GHz DDR = 2 Gbps/pin. 32 bidirectional pins → 8 bytes per clock.
  • Memory controller groups requests by DRAM row to amortize row-activation overhead. Latency goes up but throughput goes way up.
  • GPU memory is split into multiple partitions, each with its own controller; addresses interleaved across partitions in blocks of a few hundred bytes.

Streaming Cache Architecture

  • GPU working sets are huge (hundreds of MB per frame); GPU caches can't hold them.
  • CPUs aim for >99.9% hit rates; GPUs accept ~90%.
  • GPUs proceed with many misses in flight ("streaming cache") — they don't stall on a miss.
  • Texture caches must deliver enormous bandwidth (multibank/multiport SRAM) to serve many addresses per clock.

MMU & Virtual Memory

GeForce 8800: 40-bit virtual address space. Compute load/store uses 32-bit byte addresses + 40-bit offset → 40-bit virtual. MMU translates virtual to physical. Page sizes: 4–128 KB. Page table entries also specify per-page compression algorithm.

Memory Spaces in Detail

SpaceWhereNotes
GlobalExternal DRAMVisible to all CTAs in all grids. Relaxed memory ordering — same-thread same-address ordering is preserved; everything else may not be. Use bar.sync or membar for ordering.
SharedOn-chip SRAM (closely coupled to SM)16 KB per SM. 16 independently addressable banks (16-by-16 interconnect to threads). Lifetime = CTA lifetime. Doesn't compete with off-chip bandwidth.
LocalExternal DRAMPer-thread private (register spills, large stack frames). Cached on-chip.
ConstantExternal DRAM, cached in SMRead-only at kernel runtime. Cache designed to broadcast a single value to all threads in a warp per clock.
TextureExternal DRAM, streaming cacheRead-only 1D/2D/3D arrays with filtered interpolated access. Cached for throughput.

Coalescing

Local and global load/store instructions coalesce requests from the same warp into a single block transfer when addresses fall in the same block and meet alignment. This is a major performance lever.

ROP (Raster Operation Processor)

Each ROP unit paired with a memory partition. Performs depth/stencil tests, color blending, lossless color/depth compression up to 8:1, and atomic memory operations.

10.6 Floating-Point Arithmetic

Supported Formats

FormatBitsUse
Half precision16 (1s + 5e + 10m + implied bit)Storage and movement; texture filtering and pixel blending. Used in OpenEXR HDR image format.
Single precision32Most computation in programmable cores. IEEE 754-compatible.
Double precision64Newer GPUs (e.g., Tesla T10P). IEEE 754. Includes fused-multiply-add (FMA).

Multiply-Add (MAD)

Single instruction = FP multiply (truncated) followed by FP add (round-to-nearest-even). NOT fused — product is truncated before the add. Doubles instruction throughput.

FMA (Fused Multiply-Add) key

Double-precision FMA performs A × B + C with a single rounding. Full 106-bit intermediate product preserved through addition. Enables more accurate dot products, matrix multiply, polynomial eval. Also enables exact-rounded software division/sqrt — eliminating the need for HW divider/sqrt units.

Default Rounding & Quirks

  • Add/multiply use IEEE round-to-nearest-even.
  • Single-precision: input denormals flushed to sign-preserved zero; underflowed results flushed to sign-preserved zero after rounding.
  • Specific IEEE features (precise FP exceptions, denormal handling) often not implemented in early GPUs. Tesla T10P added full IEEE rounding, FMA, denormal support for double precision.

Texture-Mapping FP

GeForce 8800 Ultra delivers ~500 GFLOPS of proprietary-format FP for texture mapping, in addition to IEEE single-precision FP from the SPs.

10.7 Real Stuff: NVIDIA GeForce 8800 (Tesla)

First implementation of the Tesla unified graphics and computing architecture (Nov 2006). Supports CUDA.

Top-Level: Streaming Processor Array (SPA)

  • 128 SP cores total, organized as 16 SMs, grouped into 8 TPCs (Texture/Processor Clusters).
  • 2 SMs share a texture unit per TPC.
  • Connected to 6 × 64-bit DRAM partitions via interconnection network.
  • Host interface uses PCI-Express. Input assembler collects primitives. Work distribution dispatches threads to TPCs.

Texture/Processor Cluster (TPC)

Geometry controller + SMC + 2 SMs + texture unit. SMC arbitrates the shared texture unit, load/store path, and I/O. Texture unit is deeply pipelined with a streaming cache (handles hits and misses without stalling).

Streaming Multiprocessor (SM)

  • 8 SP cores + 2 SFUs + multithreaded instruction unit + instruction cache + read-only constant cache + 16 KB shared memory.
  • SP cores and SFUs clocked at 1.5 GHz on GeForce 8800 Ultra → 36 GFLOPS per SM.
  • Hardware-multithreaded for up to 768 concurrent threads per SM, zero scheduling overhead.
  • Schedules 24 warps per SM. Warp = 32 threads. Each issued warp executes as 4 sets of 8 threads over 4 cycles.
  • SP and SFU issue on alternate cycles → both stay busy.
  • Scoreboard + priority scheduler picks a ready warp each cycle.

SP Specs

  • 1024 scalar 32-bit registers, supports up to 96 threads.
  • FP add/multiply IEEE 754 single precision; round-to-nearest-even default. Denormal inputs flush to sign-preserved zero.
  • Implements all 32-bit and 64-bit integer arith/compare/convert/logical PTX instructions.

10.8 Real Stuff: Mapping Applications to GPUs

The book walks through several CUDA-accelerated applications. Headline takeaways:

  • N-body simulation (all-pairs): GeForce 8800 delivers >240 GFLOPS vs <2 GFLOPS on a recent sequential processor — >100× speedup. Coupled with graphical display: 16K bodies interacting at 44 fps.
  • Other domains shown in the chapter: sparse matrix solvers, sorting, searching, computational chemistry, seismic processing, physics models, computational finance.
  • Applications scale transparently: one program can run on GPUs of any size, the same one program scales to multicore CPUs as well (CUDA can target both).

10.9 Fallacies & Pitfalls core

Fallacy: GPUs are just SIMD vector multiprocessors.

No. Each GPU thread has its own scalar registers, thread-private memory, thread state, ID, independent execution and branch path, effective PC, and can address memory independently. Execution is more efficient when warp threads share a PC, but not required. A GPU is closer to MIMD with barrier synchronization and SIMT optimizations. SIMT also runs multiple warps concurrently — pure SIMD doesn't.

Fallacy: GPU performance can't grow faster than Moore's law.

No. Moore's law is just a transistor-cost rate. GPUs gain by (1) applying transistors to more parallel cores, (2) improving architecture efficiency, (3) larger chips at higher cost, (4) faster/wider memory + compression + better caches. GPUs have doubled performance every 12–18 months for ~10 years.

Fallacy: GPUs only render 3D graphics; they can't do general computation.

No. GPUs are Turing-complete. CUDA exposes the cores to arbitrary C/C++ code without going through graphics APIs.

Fallacy: GPUs cannot run double-precision FP fast.

Outdated. Modern GPUs (Tesla T10P+) support IEEE 754 double precision in hardware, currently 5–10× slower than single. Closing the gap costs incremental area and improves with each generation.

Fallacy: GPUs don't do floating-point correctly.

No. Tesla family is IEEE 754-compliant for single precision. Some specific features (denormals, precise FP exceptions) historically missing on single precision but added later. Tesla T10P adds full IEEE rounding, FMA, and denormal support for double precision.

Pitfall: Just use more threads to cover longer memory latencies.

Only works if (a) you have enough threads, and (b) the threads have good locality of memory access. Without coalesced or correlated access, the memory system itself becomes the bottleneck and even thousands of threads can't hide it.

Fallacy: O(n) algorithms are difficult to speed up.

Sometimes. PCIe transfer (~48 GB/s peak, DMA) can be the limit for low-arithmetic-intensity work like vector add. Mitigations: (1) keep data on the GPU between kernel launches, (2) overlap copy and compute (concurrent transfer + execution), (3) use CPU and GPU together as heterogeneous platform.

10.10 Concluding Remarks

  • GPUs are massively parallel processors used for graphics AND general computing.
  • Programming model: graphics API (DirectX/OpenGL) for graphics; SPMD-style CUDA for general compute.
  • Parallelism continues to scale via Moore's law (more processors per chip).
  • Only programming models that scale to hundreds of cores and thousands of threads will succeed; only applications with abundant independent parallel tasks will benefit.
  • CUDA evolving toward full C/C++. Graphics APIs adopting compute features.
  • Future: more cores, more threads, more memory bandwidth, heterogeneous CPU+GPU programming models.

Key Historical Takeaways

  • NVIDIA GeForce 3 (2001): first big step toward programmable shading.
  • ATI Radeon 9700 (2002): programmable 24-bit FP pixel fragment processor (DirectX 9).
  • XBox 360 (2005): first unified processor GPU (vertex + pixel on same processor).
  • NVIDIA GeForce 8800 (2006): first Tesla architecture GPU; first to fully support CUDA.
  • 3dfx SLI (1998): first multi-GPU scaling. NVIDIA continued with multi-GPU SLI starting GeForce 6800 (2004).

Common Exam Traps

  • "A GPU is just a fancy SIMD unit." No. SIMT lets each thread branch independently. Different warps run different code paths simultaneously without efficiency loss.
  • "Threads in a warp can communicate freely." Implicitly yes via shared memory + barriers, but only inside the same block. Threads in different blocks cannot synchronize within a kernel — must launch a new kernel or use atomics on global memory.
  • "All threads in a block must execute the same instruction at the same time." Only at the warp level (32 threads). Threads in the same warp execute together; threads in different warps of the same block may be at different points in the code.
  • "GPU caches are large like CPU caches." No. GPU caches are tiny "streaming caches." Latency is hidden by multithreading, not by hit rates.
  • "Global memory ordering is sequentially consistent." No. CUDA uses a relaxed ordering model. Within a thread, same-address ordering holds; everything else needs bar.sync or membar.
  • "MAD and FMA are the same instruction." No. MAD truncates the product before the add. FMA performs the multiply-add with a single final rounding using the full intermediate product.
  • "GPUs always use single precision." No. Modern GPUs (Tesla T10P+) support IEEE 754 double precision in hardware, including FMA and denormals.
  • "All shared memory accesses are conflict-free." No. Shared memory has ~16 banks. If two threads in the same warp access different addresses on the same bank, accesses serialize.
  • "You can do recursion in a CUDA kernel." No (in the era covered). Recursion is disallowed because providing stack space for thousands of threads would be prohibitive.
  • "GPU divergence is always cheap because the warp re-converges." No. During divergence, the inactive lanes idle. A perfectly even if-else gets you 50% of peak performance for that block.
  • "More threads is always better." No. More threads need more registers; if your per-thread register demand is high, you can't fit as many threads → less latency hiding. The compiler trades the two.
  • "PCIe bandwidth is the same as the GPU's internal memory bandwidth." No. PCIe is ~48 GB/s peak (DMA). GPU on-board memory bandwidth is many times higher (often >100 GB/s). PCIe is the bottleneck for low-arithmetic-intensity workloads.
  • "CUDA programs only run on GPUs." No. The CUDA programming model is also applicable to multicore CPUs — NVIDIA's CUDA toolchain can target both.
  • "GPU = warp-wide SIMD with no scalar registers." No. Each thread has its own scalar registers in the SP's register file. The instruction is broadcast across the warp, but the operand state is per-thread.
  • "Blocks within a grid execute in order from block 0 upward." No. CUDA explicitly says blocks may execute in any order, in parallel or in series. Programs must not assume any block ordering.
