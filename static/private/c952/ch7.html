7.1 Introduction: Principle of Locality core

Programmers want unlimited fast memory. The chapter creates that illusion via a memory hierarchy — multiple levels with different speeds and sizes. Faster memory = more expensive per bit = smaller.

Two Kinds of Locality key

Temporal localityIf a location is referenced, it tends to be referenced again soon. Source: loops re-read instructions and data.
Spatial localityIf a location is referenced, locations with nearby addresses tend to be referenced soon. Source: sequential instruction fetch, array traversal.

Memory Hierarchy Vocabulary

Memory hierarchyMultiple memory levels; as distance from processor grows, both size and access time grow.
Block (or line)Minimum unit of information that can be present or absent in a cache. Data are copied between adjacent levels in blocks, not bytes.
HitRequested data found in the upper (faster) level.
MissRequested data NOT found; lower level must supply it.
Hit rateFraction of accesses that hit. Often called hit ratio.
Miss rate1 − hit rate.
Hit timeTime to access the upper level INCLUDING the time to determine hit/miss.
Miss penaltyTime to fetch a block from the lower level into the upper level + deliver to the requestor. Hit time ≪ miss penalty.

Data are inclusive: each upper level is a subset of the level below. Data movement happens between two ADJACENT levels at a time.

7.2 Memory Technologies core

Four Primary Memory Technologies (fastest → slowest)

TechnologyWhere usedVolatile?Notes
SRAM (Static RAM)CachesYes (needs power)Fast (~1–10 ns), 4–6 transistors/bit, expensive
DRAM (Dynamic RAM)Main memoryYes (needs refresh)Slow (~50 ns), 1 transistor + capacitor/bit, cheap. RAS/CAS two-stage decode.
FlashMobile secondary, SSDNoWears out — uses wear leveling. Has replaced disks in many mobile devices.
Magnetic DiskServer secondary storageNoMechanical — milliseconds. Cheapest per bit.

DRAM Details

  • Refresh — capacitor charge leaks; must read+write each row periodically (~1–2% of cycles).
  • RAS / CAS — Row Access Strobe + Column Access Strobe; two-stage decode reusing address pins.
  • SDRAM / DDR — synchronous DRAM; DDR transfers on both rising and falling edges → 2× bandwidth.
  • Modules called DIMMs packaged with multiple DRAM chips. Named by bandwidth (e.g., PC25600).

Magnetic Disk Anatomy

TrackOne of thousands of concentric circles on a disk surface.
SectorSegment of a track. Smallest addressable unit on disk.
SeekPositioning the read/write head over the proper track.
Rotational latency / delayTime for the desired sector to rotate under the head.

Disk access time = seek time + rotational latency + transfer time + controller overhead.

7.3 The Basics of Caches core

"Cache" = safe place for hiding things. In computing it's the level closest to the processor, faster than main memory.

Direct-Mapped Cache key

Each memory block maps to exactly one cache location. Mapping is by modulo:

cache_block_index = (block_address) mod (number_of_blocks_in_cache)

Address Decomposition

For a direct-mapped cache with 2n blocks of 2m words each:

FieldBits
Byte offset (within word)2 (for 4-byte words)
Block offset (which word in block)m
Index (which cache block)n
Tag (rest, identifies which memory block)address_size − n − m − 2

Cache Entry Bookkeeping

TagThe high-order address bits stored alongside the cache block to identify which memory block the entry currently holds.
Valid bit1 = the cache entry contains a valid block; 0 = entry is empty (e.g., right after power-on).
Dirty bitFor write-back caches: 1 = the block has been modified since loaded, must be written back on eviction.

Block Size Tradeoff

  • Larger blocks exploit spatial locality → lower miss rate (up to a point).
  • Beyond a certain size, miss rate INCREASES (more pollution, fewer blocks fit).
  • Larger blocks → larger miss penalty (longer transfer). Mitigations: early restart, critical-word-first.

Write Policies key

PolicyHow writes updateProsCons
Write-throughWrites update BOTH cache and next-lower level on every write.Cache and memory stay consistent. Simple miss handling.Every write touches memory → slow. Mitigation: write buffer.
Write-backWrites update only the cache. Modified block is written to memory ONLY on eviction.Multiple writes to a block coalesce. Lower memory bandwidth.Eviction is more expensive. Need a dirty bit.

Write-Miss Handling

  • Write-allocate: on a write miss, load the block into the cache, then perform the write.
  • No-write-allocate: on a write miss, write directly to memory; don't load the block.
  • Common pairing: write-back + write-allocate; write-through + no-write-allocate.

Split vs Unified Cache

Split cacheSeparate I-cache and D-cache. Avoids structural hazard between instruction fetch and data access. Used in L1 of nearly all modern CPUs.
Unified cacheOne cache holds both instructions and data. Common at L2/L3.

Example: Intrinsity FastMATH

16 KiB I-cache and 16 KiB D-cache, each 256 blocks × 16 words/block. Used as a teaching example.

7.4 Cache Performance & Associativity core

Performance Equations key

CPU time = (CPU cycles + Memory-stall cycles) × Cycle time Memory-stall cycles = Memory accesses × Miss rate × Miss penalty = Instructions × (Misses / Instruction) × Miss penalty Average Memory Access Time (AMAT) = Hit time + Miss rate × Miss penalty

Three Cache Organizations

OrganizationBlock placementTag compares per access
Direct-mappedExactly one location1
n-way set-associativeAny of n locations within the chosen setn (in parallel)
Fully associativeAny location in the cacheAll entries (in parallel)

Set-Associative Address Decomposition

FieldUsed for
TagIdentify the block
IndexSelect the set (smaller than direct-mapped index — 1 bit per associativity-doubling)
Block offsetWord within block

As associativity increases, index shrinks and tag grows. A fully associative cache has NO index — only tag and offset.

Replacement Policy

  • LRU — Least Recently Used. Replace the block that has been unused the longest. Best miss rate, expensive at higher associativity.
  • Random — pick a candidate randomly. Cheap, almost as good as LRU at high associativity.
  • FIFO — first in, first out. Approximation to LRU.

Multilevel Caches key

Multilevel cacheHierarchy of caches (L1, L2, L3) before main memory. L1 optimized for hit time; L2/L3 optimized for miss rate.
Global miss rateFraction of all references that miss in ALL levels of the cache hierarchy.
Local miss rateFraction of references TO ONE LEVEL that miss in that level. Higher than global because it sees only the misses from above.

Designs often increase associativity for L1 hit time and increase block size for L2 to reduce miss rate.

7.5 Dependable Memory Hierarchy

Reliability vs Availability

MTTFMean Time To Failure — average time a system runs before failing.
MTTRMean Time To Repair — average time to restore service after a failure.
MTBFMean Time Between Failures = MTTF + MTTR.
AFRAnnualized Failure Rate = fraction of disks expected to fail per year. Field AFR (~2–4%) often higher than spec.
AvailabilityMTTF / (MTTF + MTTR). Improve by increasing MTTF or decreasing MTTR.

Three Strategies to Increase MTTF

  1. Fault avoidance — prevent fault occurrence by construction.
  2. Fault tolerance — use redundancy to keep meeting the spec even when components fail.
  3. Fault forecasting — predict and replace failing components before they fail.

Error Codes key

CodeCapabilityCost (overhead)
1-bit parity (distance-2)Detect any 1-bit error1 bit per word
Hamming SEC (distance-3)Single Error Correct~7 bits / 64-bit word
SEC-DED (distance-4)Single Error Correct + Double Error Detect8 bits / 64; 9 bits / 128
ChipkillSurvive entire DRAM chip failureRAID-like parity across chips

7.6 Virtual Machines

Virtualization layer that lets multiple OSes share hardware. Originally mainframe (1960s); now ubiquitous in datacenter.

Virtual Machine Monitor (VMM / hypervisor)

Software layer that presents virtual hardware to guest operating systems. Required hardware features for efficient VMs:

  • At least two processor modes (system/supervisor and user).
  • Privileged instructions only available in system mode.
  • Mechanism for the VMM to handle guest-attempted privileged operations.

x86 has 18 instructions that historically caused virtualization problems (e.g., POPF, PUSHF reading a flag that should trap but doesn't). Hardware extensions (Intel VT, AMD-V) added to fix this.

7.7 Virtual Memory core

"Virtual memory uses main memory as a cache for secondary storage." Same hierarchy concepts as caches; different vocabulary.

Vocabulary key

Virtual addressAddress the program uses; translated by hardware to a physical address.
Physical addressAddress in main memory.
Address translation / mappingProcess of mapping a virtual address to a physical address.
PageUnit of virtual memory transfer (analogous to a cache block). Typically 4 KiB (or larger).
Page faultAn accessed page is not present in main memory; OS must fetch it from disk (huge miss penalty: millions of cycles).
ProtectionMechanisms ensuring multiple processes sharing the processor/memory/I/O can't interfere with each other.
SegmentationVariable-size mapping (segment + offset). Older alternative to paging.

Page Table

  • Indexed by virtual page number; entries hold the physical page number plus permission/valid/dirty/reference bits.
  • Each process has its own page table. Page Table Register holds its base address.
  • Swap space = disk area reserved for the full virtual memory of a process.
  • Reference bit / use bit / access bit — set whenever the page is accessed; OS uses for LRU approximation.
  • ARMv8 uses 4-level page tables to translate a 48-bit virtual → 40-bit physical address.

Translation Lookaside Buffer (TLB) key

Cache of recent virtual→physical translations. Without it, every memory access would need a page-table lookup (one or more memory accesses just to translate).

  • Typical TLB: 16–512 entries, 1–2 page-table entries per "block," miss penalty 10–100 cycles, hit time < 1 cycle.
  • TLB miss can mean (a) page is in memory but not in TLB → just load entry; (b) page is not in memory → page fault.
  • Combined with the data cache: virtual address → TLB → physical address → cache lookup.

Virtually vs Physically Addressed Caches

Virtually addressed cacheIndexed by virtual address. Faster (no translation before lookup) but suffers from aliasing: two virtual addresses can map to the same physical address.
Physically addressed cacheIndexed by physical address. No aliasing. Translation must happen first (or in parallel via virtually-indexed-physically-tagged designs).

Protection Vocabulary

Supervisor / kernel modeMode in which a process (the OS) can execute privileged instructions.
System callSpecial instruction that transfers control from user mode to a defined location in supervisor mode.
Context switchSave current process's state (registers, PC, page-table base) and load another's. OS-orchestrated.
Restartable instructionAn instruction that can resume after an exception is resolved (essential for recovering from page faults).

Virtual Memory Design Choices to Reduce Miss Rate

  • Make pages large (4 KiB → 2 MiB / 1 GiB) — more spatial locality per page.
  • Fully associative placement (any virtual page → any physical page) — eliminates conflict misses.
  • OS handles page replacement (LRU approximation via reference bits).
  • Write-back only — writing to disk on every store would be catastrophic.

7.8 A Common Framework for Memory Hierarchy core

All hierarchy levels (L1, L2, L3, virtual memory) share a common design framework around four questions:

The Four Questions

QChoices
1. Where can a block be placed?One place (direct mapped) | A few places (set associative) | Any place (fully associative)
2. How is a block found?Indexing (direct) | Limited search of a set (set assoc.) | Full search (fully assoc.) | Lookup table (page table for virtual memory)
3. Which block to replace on a miss?LRU | Random | FIFO
4. What happens on a write?Write-through (with or without write-allocate) | Write-back (always write-allocate)

Three Cs Model key

An intuitive classification of all cache misses:

TypeCauseHow to reduce
Compulsory (cold-start)First access to a block — block has never been in the cacheLarger blocks (more spatial locality), prefetching
CapacityCache is too small to hold all blocks needed during execution, even with full associativityLarger cache (or use the lower level)
Conflict (collision)In a set-associative or direct-mapped cache, multiple blocks compete for the same setHigher associativity (or larger cache)

A fully associative cache has NO conflict misses by definition.

7.9 FSM Cache Controller

Build cache control as a finite-state machine. Simple cache: write-back, direct-mapped.

FSM States

StateAction
IdleWait for a valid read or write request from the processor → move to Compare Tag.
Compare TagTest whether the request hits or misses in the cache. Hit → service request (deliver data or write), back to Idle. Miss → check dirty bit.
Write-BackWrite the dirty block back to memory using {tag, index}. Wait for Ready.
AllocateFetch the new block from memory. Wait for Ready, then re-attempt by going to Compare Tag.

7.10 Cache Coherence core

In a multicore system, multiple caches may hold copies of the same memory location. The cache coherence problem arises when one core writes a value others have cached.

Coherence Properties

  • Coherence defines what value a read returns: a read after a write to the same location should return the new value (eventually).
  • Consistency defines when a write becomes visible to reads — orderings between operations on different addresses.

Migration & Replication

MigrationA data item moves to a local cache and is used there. Reduces latency to access shared data and bandwidth on shared memory.
ReplicationMultiple caches hold copies of shared read-mostly data. Reduces latency and contention for reads.

Snooping Protocols

Each cache controller "snoops" the shared bus, watching for transactions on blocks it caches. Two main styles:

StyleHow it works
Write-invalidateOn write, broadcast an invalidate to all other caches. Only one writer at a time. Most common.
Write-updateOn write, broadcast the new value to all caches that hold the block.

False Sharing key

Two unrelated shared variables happen to lie in the same cache block. Each write to one invalidates the entire block in other caches, even though those caches care only about the OTHER variable. Hurts performance with no semantic conflict. Mitigation: pad/align variables onto distinct cache lines.

MSI Protocol States

StateMeaning
InvalidBlock is not present (or has been invalidated).
SharedBlock is present and may be shared by other caches; matches memory.
ModifiedBlock is present and has been modified (dirty); no other cache has it; differs from memory.

7.11 Redundant Arrays of Inexpensive Disks (RAID)

Group of small inexpensive disks treated as one. Improves bandwidth (striping), availability (mirroring/parity), and density.

Key Concepts

StripingSpread sequential blocks across multiple disks → parallel access → higher bandwidth.
MirroringIdentical data written to multiple disks → high availability + read parallelism.
Protection groupSet of data disks/blocks sharing a common check disk/parity block.
Hot swappingReplace a hardware component while the system is running.
Standby sparesReserve hardware that can replace a failed component immediately.

RAID Levels

LevelSchemeNotes
RAID 0Striping only, no redundancyPerformance only — not real RAID
RAID 1Mirroring2× capacity cost. Widely used in servers.
RAID 2Hamming codeNot used in practice
RAID 3Bit-interleaved parityOne parity disk; rare
RAID 4Block-interleaved parityOne parity disk → bottleneck on writes
RAID 5Distributed block-interleaved parityParity rotates across all disks → no single bottleneck. Widely used.
RAID 6P + Q double paritySurvives 2 simultaneous disk failures

RAID 1 and RAID 5 are widely deployed in servers.

7.13 Real Stuff: ARM Cortex-A53 & Intel Core i7

ARM Cortex-A53 Memory Hierarchy

  • L1: split I + D, 16–64 KiB each, 2-way associative.
  • L2 unified, 128 KiB – 2 MiB, 16-way associative.
  • 40-bit physical address.
  • Two-level TLB (micro-TLB + main TLB), 4–512 entries.

Intel Core i7 920 Memory Hierarchy

  • L1: split, 32 KiB I + 32 KiB D, 8-way associative.
  • L2 unified per core, 256 KiB, 8-way.
  • L3 unified shared, 8 MiB, 16-way (inclusive of L1/L2).
  • Two-level TLB.
  • 48-bit virtual / 36-bit physical address.

Nonblocking Cache key

A cache that allows the processor to continue making memory references while the cache is handling earlier misses. Essential for OOO superscalar CPUs — without it, a single L1 miss would stall the whole pipeline.

7.15 Going Faster: Cache Blocking & Matrix Multiply

For matrices that don't fit in the cache, naive matrix multiply suffers because each iteration of the outer loop refetches the entire B matrix. Solution: cache blocking (a.k.a. tiling).

Cache Blocking Idea

Process the matrix in BLOCKSIZE × BLOCKSIZE submatrices. Within a block, all data fit in L1 and are reused. Pseudo:

for (sj = 0; sj < n; sj += BLOCKSIZE)
   for (si = 0; si < n; si += BLOCKSIZE)
      for (sk = 0; sk < n; sk += BLOCKSIZE)
         do_block(n, si, sj, sk, A, B, C);

Speed-up

Cache blocking + the SIMD/ILP ideas from earlier chapters give 2–2.5× more on top of the prior baseline; combined with multicore (Ch8), the total speed-up over unoptimized C is ~212× for 960×960.

7.16 Fallacies and Pitfalls

Pitfall: Ignoring memory system behavior in code/compiler.

Cache-oblivious code can be 10× slower than cache-aware code on the same hardware. Loop ordering, blocking, alignment matter.

Pitfall: Forgetting that sequential access is special.

Disk and DRAM both reward sequential access patterns. Random access can be 100× slower for disks, 5× slower for DRAM (different rows).

Fallacy: Disk failure rates in the field match their specifications.

Spec MTTF often translates to AFR ~0.5–1%. Field measurements (Google, others) show 2–4% AFR — 2–4× worse than spec.

Pitfall: Implementing a virtual machine monitor on an ISA not designed for VMs.

x86 has 18 instructions whose behavior is wrong for virtualization (e.g., POPF should trap when guests change the interrupt-enable bit, but it silently ignores the change). Intel VT and AMD-V added trap-on-execute hardware to fix this.

Prefetching

Hardware-initiated technique: speculatively bring data into the cache before it's referenced, using known stride patterns or program hints. Helps cover compulsory misses but wastes bandwidth on bad guesses.

Common Exam Traps

  • "Hit time and miss penalty are the same thing." No. Hit time = time to access the upper level (small). Miss penalty = time to bring a block from the LOWER level (much larger).
  • "AMAT = Miss rate × Miss penalty." No. AMAT = HitTime + MissRate × MissPenalty. Hit time is paid on every access.
  • "Larger blocks always reduce miss rate." Up to a point. Beyond a sweet spot, the cache holds fewer blocks and the miss rate rises again.
  • "Write-back doesn't access memory until eviction, so it's always faster." Mostly, but eviction is more expensive (must write the dirty block) and you need a dirty bit per block.
  • "Write-through means the CPU stalls on every write." No, only if there's no write buffer. With a write buffer, the CPU continues; the buffer absorbs the writes.
  • "Direct-mapped caches don't have tags." They do. The tag identifies WHICH of the many memory blocks that map to a slot is currently stored there.
  • "A fully associative cache has no misses." No. It has no CONFLICT misses, but it still has compulsory and capacity misses.
  • "LRU is the only sensible replacement policy." No. Random and FIFO are common; at high associativity, Random is nearly as good and much cheaper.
  • "All cache misses cost the same time." No. An L1 miss that hits in L2 costs ~10s of cycles; an L2 miss that hits in L3 costs ~50; main memory ~200; disk ~millions.
  • "Virtual memory is just paging." Mostly true today, but historically segmentation was an alternative; many systems used both. The exam may ask about both.
  • "A TLB miss is the same as a page fault." No. A TLB miss might just mean the translation isn't cached — the page might still be in memory. A page fault means the page isn't in memory at all → millions of cycles.
  • "Page tables are stored in special memory." No. Page tables live in main memory; the Page Table Register holds the base address. The TLB caches recent entries.
  • "A virtually-addressed cache is always faster than a physically-addressed one." Faster on a hit, but vulnerable to aliasing (two virtual → one physical) which can produce stale data. Most modern designs use virtually-indexed-physically-tagged to get the best of both.
  • "False sharing is a bug in the program." No. The program is logically correct but slow because two unrelated variables happen to share a cache block. Fix by padding/aligning to separate cache lines.
  • "RAID 0 is real RAID." No. RAID 0 has NO redundancy — it only stripes. The "R" stands for Redundant; RAID 0 doesn't qualify (and increases the chance of data loss vs a single disk).
  • "Cache coherence and memory consistency are the same thing." No. Coherence: all caches eventually agree on the value at one address. Consistency: ordering rules among accesses to DIFFERENT addresses.
  • "Compulsory misses can be eliminated entirely." Reduced, not eliminated. Larger blocks and prefetching help, but the very first access to a block is unavoidable without a perfect oracle.
  • "Disk failure rates measured in the field match the spec." No. Field studies (Google, etc.) show 2–4% AFR vs 0.5–1% in spec — 2–4× worse.
  • "Three Cs: capacity, conflict, coherence." No. Compulsory, capacity, conflict. (Coherence is sometimes added as a "fourth C" in multicore contexts but isn't part of the original model.)
  • "x86 was always virtualization-friendly." No. x86 had 18 instructions that misbehaved under naive virtualization. Intel VT and AMD-V added hardware to fix this.
