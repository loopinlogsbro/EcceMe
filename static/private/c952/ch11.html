11.1 Introduction core

Control has two parts: a combinational piece (no state — pure logic from inputs to outputs) and a sequential piece (state machine that walks through cycles). Single-cycle designs use only combinational control; multicycle designs need a sequential controller.

Combinational controlStateless. Inputs (e.g., opcode) → outputs (control signals) in one pass through gates. Used for single-cycle datapath and the ALU control block.
Sequential control unitStateful. Walks through multiple states per instruction (multicycle). Implemented as an FSM or microprogram.
FSM-based controlNext-state logic computed from current state + opcode. Initial form: state diagram.
MicroprogramEach microinstruction = one state. Initial form: a "program" of microinstructions in a control store (ROM/PLA).
Two next-state strategies(1) Encode the next-state function in logic. (2) Use a counter that increments by default + branch logic for non-sequential transitions.

Reference: COD Appendix C sections C.2–C.5. zyBook uses MIPS instead of LEGv8 here.

11.2 Implementing Combinational Control Units core

Map a truth table directly to gates (AND/OR network) or to a structured two-level array (PLA). Modern CAD tools do this automatically and exploit don't-care terms to minimize logic.

ALU Control Block

  • 4 outputs: Operation3, Operation2, Operation1, Operation0.
  • Inputs: 2-bit ALUOp + 6-bit function code (only 4 of the 6 function bits actually matter — upper 2 are don't cares).
  • Operation3 has no truth-table row because it is only set for the NOR operation, which is always 0 in this subset.

ALU Operation Encoding

ALU Control (Op3..Op0)Function
0000AND
0001OR
0010Add
0110Subtract
0111Set on less than (slt)
1100NOR

Programmable Logic Array (PLA) key

StructureTwo-level: an AND plane (computes minterms = product terms from inputs and inverses) followed by an OR plane (sums minterms into outputs).
Why PLA over gatesRegular structure, easy to size, only encodes the minterms that are actually used (sparse) — much smaller than a full ROM if only a handful of input combos are active.
Don't caresInputs that may be 0 or 1 without affecting output. Used by minimization tools (e.g., Espresso) to merge truth-table rows and reduce gate count.
When the main control fitsUnstructured gate logic is fine for small control functions. Once you have many opcodes and many control lines, use a PLA or ROM instead.

11.3 Implementing Finite-State Machine Control core

Multicycle MIPS control has 10 states. Encode each state as a 4-bit number (S3..S0) stored in a state register that's written on the active clock edge. The control logic is combinational and produces (a) datapath control signals (depend only on current state) and (b) next-state outputs NS3..NS0 (depend on current state + opcode).

State registerHolds current-state bits. Written on clock edge, stable for the rest of the cycle.
NS outputsNext-state bits the control logic emits; clocked into the state register and become the new current state next cycle.
Datapath control signalsDepend only on current state (not opcode). Same combination of asserts in every cycle that visits a given state.
Next-state functionDepends on current state + 6-bit opcode. Most logic complexity lives here.

Two Implementation Choices for the Truth Table

ImplementationNotesSize for this MIPS subset
ROM (single)10 inputs (6 opcode + 4 state) → 1024 entries × 20 bits/word~20 Kbits — wasteful (lots of duplicates and don't-cares)
ROM (split into two)One ROM for datapath signals (16 bits, only 4 inputs = state), one ROM for next-state (4 bits, 10 inputs). Datapath part doesn't depend on opcode.~4.3 Kbits — about ⅕ of single ROM
PLA (single)17 unique minterms × (10 inputs + 20 outputs)~510 PLA cells — much smaller than ROM
PLA (split into two)Datapath PLA + next-state PLA~298 cells (≈55% of single PLA)

For multicycle MIPS, ~94% of the ROM bits are devoted to the next-state function — this is the wasteful part PLAs (or sequencers, §11.4) compress.

Building Logic Equations from the State Diagram

  • For each output, OR together every term that asserts it.
  • "+" means OR, juxtaposition means AND.
  • stateN is the AND of state-bit conditions encoding state N.
  • (Op = 'lw') is the AND of opcode bits encoding lw.
  • NextStateN drives NS[3:0] to the binary value N.

Example: Equation for NS0

NS0 = NextState1 + NextState3 + NextState5 + NextState7 + NextState9 (NS0 must be 1 in any state whose binary number ends in 1.)

11.4 Implementing the Next-State Function with a Sequencer core

Most states fall through to state+1. So replace the explicit next-state logic with: counter (increment current state) + small dispatch table(s) for branches. Saves enormous ROM/PLA area when state count is large.

AddrCtl Encoding (2-bit field per microinstruction)

AddrCtlAction
0Set state to 0 (Fetch — next instruction)
1Dispatch with ROM 1 (state 1 → branch by opcode class)
2Dispatch with ROM 2 (state 2 → branch lw vs sw)
3Use the incremented (sequential) state — Seq

Microprogram / Microcode Vocabulary key

MicroinstructionOne control word — i.e., one state's worth of control signals + sequencing field.
MicroprogramThe full set of microinstructions implementing the ISA's control.
Microprogram counter (μPC)The state register, viewed as a counter pointing at the current microinstruction in control store.
Control storeThe ROM/PLA holding microinstructions.
Dispatch ROMSmall lookup table that maps an opcode to the next microinstruction address. Used for branching from a "decode" state.

Logic-Reduction Techniques

Logic minimizationUse don't-cares + algebraic manipulation to reduce gate/term count. Best left to a CAD tool.
State assignmentChoose state numbers so logic equations contain redundancy. E.g., renumbering states 4 and 7 to 8 and 9 lets RegWrite = S3 — one bit test instead of two row equations. Limited when an explicit counter forces state ordering.

11.5 Translating a Microprogram to Hardware core

To turn a symbolic microprogram into a binary control store: (1) define how each microinstruction field encodes into control signals, (2) assign addresses to microinstructions, (3) fill in the dispatch ROMs.

Sequencing Field Values

SymbolAddrCtlMeaning
Fetch0Go to Fetch state (state 0)
Dispatch 11Branch via dispatch ROM 1 (opcode-driven)
Dispatch 22Branch via dispatch ROM 2 (opcode-driven)
Seq3Increment μPC (sequential next state)

Default-Value Rule

If a microinstruction field controls a stateful unit (Memory, ALU dest, PCWriteControl, etc.) and is left blank, no control signal is asserted. If a field controls a mux or ALUOp and is blank, the output is don't-care (associated signals don't matter).

Reducing Microinstruction Width key

  • Encoding: If only one of N control lines is ever asserted at a time, encode them into a ⌈log₂N⌉-bit field with a decoder. Trades width for a small decoder + signal delay. (e.g., 8 → 3 bits saves 5 bits per microinstruction.)
  • Format fields (opcode-style): Multiple microinstruction formats (e.g., one for memory ops, one for ALU). Default values fill in unspecified lines. Cuts width but may need extra microinstructions.

Horizontal vs Vertical Microcode core

Horizontal microcodeWide microinstructions, few fields encoded. One μinst can drive many parallel datapath actions. Fewer cycles per ISA instruction; larger control store.
Vertical microcodeNarrow, heavily encoded microinstructions. More μinsts per ISA instruction; smaller control store.
Better terminology"Maximally encoded" (vertical) vs "minimally encoded" (horizontal). The horizontal/vertical labels are not universally defined — Intel called the 8086's 21-bit μinst "horizontal" relative to peers.

11.6 Concluding Remarks

Whether you express control as a state diagram or a microprogram, the hardware translation is the same: each state asserts control outputs and chooses the next state.

Big Picture Tradeoffs

  • Encoded next-state vs explicit sequencer: sequencer wins when the FSM has many states with long sequential runs and few branches.
  • ROM vs PLA: PLA wins unless the control function is very dense (most input combinations meaningful). ROM may win if control sits on a separate chip.
  • Translation flexibility: A microcode assembler has freedom in microinstruction address assignment (only constraint is that sequential μinsts must be sequential in memory).

Common Exam Traps

  • "All control logic is combinational." No. Single-cycle controllers are; multicycle controllers must be sequential (have a state register).
  • "A PLA and a ROM are the same thing in different packaging." No. A ROM has one entry per input combination (full truth table); a PLA encodes only the minterms actually used. PLAs are usually much smaller for sparse control functions.
  • "Splitting a ROM cannot save space — it's the same data." No. If part of the function depends only on a subset of the inputs (datapath signals depend only on state, not opcode), you can store that part with far fewer entries.
  • "An FSM and a microprogram are different mechanisms." No. They are alternative representations of the same control function. A microprogram can be implemented with an FSM, ROM, PLA, or sequencer.
  • "With an explicit sequencer, you no longer need a state register." No. The sequencer's counter is the state register — it is now called a microprogram counter (μPC).
  • "Encoding control lines into a smaller field always speeds the machine up." No. Encoding adds a decoder in the signal path (a small delay) and may require an extra microinstruction when two encoded lines must both be active.
  • "Horizontal microcode has more bits, so it's always more powerful." Misleading. Wider μinsts allow more parallel datapath ops per cycle (lower CPI), but if the parallelism isn't used, the wider control store is wasted area.
  • "Don't-cares are bugs." No. They are a feature: they tell the minimization tool that a row may be merged with adjacent rows, shrinking the gate count.
  • "NextState1 means the next clock cycle the state will be 1, regardless of opcode." No. NextState1 is asserted only in the rows of the next-state truth table where state 1 should follow — it's conditional on the inputs of that row.
  • "The dispatch ROM holds control signals." No. It holds microinstruction addresses (state numbers) — the next-state target for an opcode-driven branch.
